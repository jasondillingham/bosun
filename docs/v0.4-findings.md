# v0.4 round-1 findings (kickoff failures)

Field notes from attempting to launch the v0.4 round on 2026-05-16.
Not blockers for v0.4 *features* — these are bosun-the-tool gaps
surfaced by the kickoff itself. File for a future round to fix.

## Bug 1: `bosun init` hangs silently when `git worktree add` stalls

**Symptom.** `bosun init 10 --brief …` produced exactly one line of
output ("Created 0 session(s)…" — actually nothing) and stayed silent
for 5+ minutes. `ps` showed `git worktree add` running with 0.02s
CPU time. The on-disk worktree directory had files in it; git was
hung on something post-checkout (most likely an fsync).

**Reproducer (kind of).** Run the macOS system under heavy background
load — bulk delete of caches that triggers Spotlight reindex +
iCloud verification — then run `bosun init N`. Even N=4 hung on the
first `git worktree add`. Load average was 11+; cloudd burning
35% CPU; WindowServer at 44%; 6 stuck processes per `top`.

**Root cause (not bosun's fault, but bosun handles it badly).** APFS
fsync stalls behind whatever else the kernel is flushing. git's
worktree-add finishes the file writes but blocks on the closing
sync. There's no timeout in `internal/git/git.go` for any subprocess
invocation. bosun also prints nothing between "starting init" and
"created N sessions," so the operator can't tell hang from progress.

**What bosun should do.**

1. **Per-operation timeout** on `c.run(ctx, dir, "worktree", "add", …)`
   et al. Default 30s, configurable via `.bosun/config.json`. A
   timeout returns "git worktree add timed out after 30s — likely
   filesystem pressure; check `top` / `iostat`" rather than hanging.
2. **Progress reporting** in `cmd_init.go`. Print "Creating worktree
   session-1 (1/N)…" before each `AddWorktree` call. Operator sees
   forward progress vs. silent hang.
3. **Pre-flight system-load check.** `top -l 1` or `uptime` parsing
   to detect load > 5 (or whatever heuristic). Warn the operator
   "system load is X; init may be slow" but don't refuse.
4. **Recovery on partial init.** If init dies after creating
   worktrees 1..K of N, the operator is left with orphan branches +
   half-populated worktrees that the next `bosun init` won't reuse.
   `bosun init --resume` would re-derive what's done and continue.
   Alternatively, init could be transactional: if step X fails,
   roll back steps 1..X-1.

## Bug 2: macOS Finder phantom branches (`bosun/session-N 2`)

**Symptom.** Repeated runs leave broken-name refs in `.git/refs/heads/`:
`bosun/session-2 2`, `bosun/session-3 3`, etc. These show up as
warnings on every `git branch` and pollute `bosun status
--with-overlaps` (counted as separate sessions).

**Suspected cause.** Some macOS process (Finder? Time Machine?
Spotlight?) duplicates files in `.git/` when it encounters them
during a sweep. The "X 2" suffix is the macOS convention for
duplicate filenames.

**What bosun should do.**

1. **Detect + clean.** `bosun init` could check `.git/refs/heads/` for
   the `bosun/session-N <digit>` pattern and offer to delete them at
   start, with a `--no-clean-phantoms` opt-out.
2. **Document.** Add a `.git/refs/heads/bosun/.gitkeep`-style note in
   the README about the phantom-branch failure mode and the manual
   cleanup (`rm '.git/refs/heads/bosun/session-N N'`).
3. **Investigate root cause.** Whatever is duplicating these files is
   doing it across `~/Documents/Homelab/`. Could be Time Machine's
   Local Snapshots, Spotlight, or a finder operation. Worth one
   reproducible session to nail down.

## Bug 3: Read-only GOMODCACHE files survive `bosun remove --force`

**Already partially fixed** in v0.3.1 by session-3's chmod-before-remove
helper (`internal/git.chmodWritableTree`). But the production
`bosun-bosun-1` cleanup STILL hit Permission denied even with that
fix in place. The realistic-GOMODCACHE-shape test we wrote passes,
which means the test doesn't capture the failure mode.

**What's still needed.**

1. **Capture the actual failing tree.** When this hits again, copy
   the worktree to a side dir BEFORE the rm, then write a test that
   replays it exactly.
2. **Iterative chmod-then-retry-remove loop.** If `git worktree
   remove` fails with EACCES on a specific path, chmod just that
   subtree and retry. Bounded retries (3?).
3. **Investigate macOS file flags.** `chflags noschg` may be needed
   on top of `chmod u+w` for some Go cache contents. Worth checking
   `ls -lO` on a broken tree.

## Bug 4: Stale `BOSUN_MCP_SOCK` in operator's env wedges discovery

**Already fixed** in v0.3.1 by session-2's
`inheritedSocketBelongsToRepo` check. Documented here for
completeness — it bit the v0.3 kickoff before the fix landed.

## What we learned for future kickoffs

- Run `df -h ~` and `uptime` before any 4+-session kickoff. Free
  disk < 20G or load avg > 5 means wait.
- Avoid running bulk filesystem operations (cache cleanup, large
  deletes) immediately before a bosun init. Give APFS + Spotlight
  time to drain.
- `--launch` adds N Ghostty windows to the load picture. Skip it
  when the OS is already under pressure; use `bosun launch <session>`
  manually for the few you want to attach to.

## TODO: `bosun doctor` / `bosun preflight` subcommand

Today every "should I kick off?" decision is made by the operator
eyeballing `uptime`, `df -h`, and `top`. Bosun should own that check.

Build a `bosun doctor` (or `preflight`, `check`, `status --health`)
subcommand that surfaces a single red/yellow/green verdict plus
machine-readable JSON. Run on demand; also called implicitly before
`bosun init` with `--no-preflight` to skip.

### What it should check

**System resources** (covers the bugs we hit this round):
- Free disk space on the repo's filesystem. Red if < 5G, yellow if
  < 20G. The 18G-free state where v0.4 first stalled was already
  yellow.
- 1-min load average. Red if > N*2 where N is core count, yellow if
  > N. Loose heuristic but catches the 50.62 we just saw.
- Recent disk activity (`iostat`-equivalent). Red if a single disk is
  > 80% busy. Catches the case where load is OK but the disk queue is
  pathological.
- macOS only: a quick `mdutil -s /` check that Spotlight isn't in the
  middle of a full reindex.

**Bosun-internal state** (catches the rest of the v0.4 surprises):
- MCP daemon health: pidfile exists → process alive → socket
  accepts connections. Red if pidfile points at a dead process; yellow
  if pidfile is missing.
- Stale lock files: `.bosun/mcp.lock`, `.bosun/claims/.lock` —
  detect locks held by dead processes (POSIX flock semantics make
  this nontrivial; may need a heuristic like "lock file mtime > N min
  + no process holds it").
- `BOSUN_MCP_SOCK` env var validation: if set, does the path point
  at a socket that belongs to this repo? (The v0.3.1 fix added this
  check inside `ensureMcp`; surface it as a doctor finding too.)
- Orphan worktrees: dirs under the parent that match the suffix
  pattern but aren't in `git worktree list`. Yellow finding,
  suggest `bosun cleanup --orphan-dirs` (when round 2's session-1
  lands).
- Macos Finder phantom branch refs (`bosun/session-N N`). Yellow,
  with a one-line fix suggestion.

**External-tool sanity**:
- `git` on PATH and version >= the minimum bosun needs.
- The launcher chosen (Ghostty, iTerm2, etc.) is actually installed.

### Shape

```
$ bosun doctor
✓ disk:      91G free (79% used)
✓ load:      8.12 / 6.63 / 6.30 (12-core, healthy)
✓ mcp:       pid 82378 alive, socket healthy
! finder:    1 phantom branch ref (bosun/session-2 2) — remove with
             `rm '.git/refs/heads/bosun/session-2 2'`
x git:       not on PATH

verdict: ❌ blocked — git not on PATH
```

JSON output mirrors the `JSONSchemaVersion`-versioned shape every
other producer uses.

### Why this matters

Most of the silent failures we hit this round (silent init hang,
stale BOSUN_MCP_SOCK from another repo, phantom branch refs, lock
file from a prior crashed bosun) would surface as actionable doctor
findings before they become a 10-minute debug session. Adding it
also moves bosun from "tool that runs commands" to "tool that knows
whether it can run commands" — closer to the operator-shoulder model
the TUI is reaching for.

### Out of scope for this todo

- Automated remediation (`bosun doctor --fix` that deletes phantom
  refs, kills stale daemons, etc.). Tempting but risky — the doctor
  reports; the operator decides. v0.5+.
- Continuous monitoring (a daemon that watches health between
  invocations). The status command already shows session state;
  health is a kickoff-time question.

## TODO: `bosun cleanup --force` should not nuke DONE-but-unmerged sessions

Surfaced during the v0.5 round-1 merge recovery. After session-2's
hand-resolved squash re-conflicted (the known merge-re-conflict bug),
I ran `bosun cleanup --force` to clear out sessions 1-4 so the
dep-aware merge would unblock session-5. **The `--force` flag
bypassed the safety gate for sessions 5 and 6 too** — both were DONE
but their work was on their branches, never merged to main. The
worktrees and branches were deleted; the commits became unreachable
(recoverable via `git fsck --dangling` for ~14 days but slow).

The fundamental issue: `--force` is too coarse. Today it means "ignore
all safety gates," including the "this session has commits no one
else has" gate that exists specifically to prevent data loss. There's
no way to say "force-clean the merged-via-tree-equivalence sessions
but keep sessions whose work isn't on main yet."

### What bosun should do

1. **`cleanup --force` should still refuse DONE-but-unmerged
   sessions** by default. The flag's job is to override
   "WORKING-with-changes-pending" sessions, not to discard committed
   work. If the operator wants the latter behavior, require an
   even-louder opt-in: `--force-discard` or `--purge`.
2. **Detect the unmerged-work case explicitly.** `UnmergedPatches`
   already exists; use it in the cleanup planner. A session with
   `UnmergedPatches > 0` AND no patch-id evidence on main is
   irrecoverable data — the planner should bail unless explicitly
   told to discard.
3. **`bosun remove --force <session>` should have the same gate.**
   Same data-loss potential per-session; same fix shape.
4. **Recovery path.** When the planner detects unmerged work and
   refuses, the error should name the recovery actions: `bosun merge
   <session>` or `git branch <name> $(git fsck --dangling | grep
   commit)` for an already-deleted case.

### Why this matters

The whole point of the bosun model is that an operator can run
many parallel agents and trust the tool not to lose their work. A
flag that silently discards committed work breaks that contract,
even with `--force` semantics. It also undercuts the dogfood story:
the v0.5 round-1 incident lost ~10 minutes of agent work (recoverable
by re-running, but still a real cost in tokens and time).

## TODO: agent-crash / API-error / hung-session resilience (v0.6 anchor candidate)

Surfaced ahead of the real-repo trial: **bosun must not corrupt an
operator's codebase when an agent dies, hangs, or has its API call
fail mid-task.** Today's safety relies almost entirely on git
worktree isolation (main is unreachable from a session) plus
operator-driven merge gates. That's load-bearing but incomplete — it
protects main but doesn't help when bosun ITSELF acts on a session
that's in a bad state.

### Failure modes to design for

1. **Claude API 500 / network drop mid-call.** Agent's reasoning step
   fails, control returns to the user but the worktree state is
   whatever it was when the call dropped — possibly mid-edit, mid-
   commit. Bosun sees the session as still "WORKING."
2. **Agent process crashes / SIGKILLed / window force-closed.** Same
   end state. Bosun's RUNNING column already detects "no process
   alive in worktree" (v0.3 work) but doesn't act on it.
3. **Agent hangs / wedged tool call.** Process is alive, doing
   nothing. Bosun has no heartbeat — looks identical to "working."
4. **Operator reboots, sleeps, or upgrades macOS mid-round.**
   Everything dies at once. State files survive; running daemons
   don't.
5. **Disk full or OOM mid-write.** Torn files in a worktree.
   `git status` may still say "clean" because the truncation
   happened to an untracked file or an in-flight write.
6. **Operator force-closes the bosun init process** while it's
   creating worktrees. Half-populated dirs + half-created git
   metadata. We've actually seen this twice in dogfood.
7. **MCP daemon crashes mid-session.** Agent's `bosun_claim` /
   `bosun_done` calls start failing silently; agent doesn't notice;
   bosun status shows stale claims.
8. **Hook command hangs.** Pre/post-merge or pre-cleanup hook runs
   forever. Bosun blocks waiting on it.

### What bosun already protects against

- **main is never directly edited** by an agent — they live in
  worktrees, isolated by git's worktree model. This is the
  biggest single protection and it's solid.
- **Squash merges are atomic from `git`'s perspective.** A torn
  merge leaves the index conflicted but doesn't corrupt main.
- **Cross-process flocks** on claims + state (v0.3.1, v0.5 r2) —
  concurrent writers don't tear JSON.
- **The `--purge` gate** (v0.5 r2) — `cleanup --force` no longer
  silently discards committed work.
- **MCP daemon pidfile + socket validation** (v0.3.1) — stale
  daemons get detected and replaced rather than reused.
- **Tree-equivalence in merge** (v0.3.1) — a hand-resolved squash
  is recognized as already-merged on the next attempt.
- **Per-op git timeouts** (v0.5 r2) — `git worktree add` won't
  hang forever on system pressure.

### What's missing — the v0.6 ask

1. **Agent-liveness gate on destructive bosun ops.** `bosun merge`,
   `bosun remove`, `bosun cleanup` should refuse (or require a
   `--ignore-running` flag) when the session's worktree has a live
   agent process AND uncommitted dirty files. Today the operator
   can `bosun merge` a session whose agent is mid-edit; the unstaged
   changes are silently dropped from the squash. Use `internal/proc`'s
   `Running()` (already exists). Same gate for `bosun cleanup` on a
   session whose worktree is dirty AND has a live agent.
2. **Heartbeat / staleness detection.** New MCP tool
   `bosun_heartbeat()` the agent calls every N seconds (or
   piggybacks on existing tool calls). Status records last-seen
   timestamp; a session with no activity for >5 min while marked
   RUNNING is surfaced as **STALE** (new state). The TUI lights
   it yellow; CLI surfaces a "session-X appears stalled — check
   manually" line. Doesn't take destructive action; just warns.
3. **Hook timeout enforcement.** `hooks.Hook` already has a
   `TimeoutSeconds` field — verify it's actually enforced in
   `hooks.Run`. Default to 30s if zero, never zero-means-infinite.
   A hung pre-merge hook should fail with a clear "hook timed out"
   error, not block bosun forever.
4. **Pre-merge worktree integrity check.** Before `git merge
   --squash <session>`, run `git -C <worktree> fsck --no-progress`
   (cheap on a small repo, ~50ms). If it errors, refuse the merge
   with the fsck output. Catches the torn-write case.
5. **Reflog-based undo for merges.** `bosun merge` records the
   pre-merge main HEAD in `.bosun/merges.log` (timestamp, session,
   pre-SHA, post-SHA). A new `bosun merge --undo <merge-id>` resets
   main to the pre-merge SHA + restores the session state file. Not
   technically necessary (git reflog already covers this) but the
   operator who just lost an hour of work doesn't want to learn
   git reflog under stress.
6. **Init resumability.** `bosun init` writes per-step state into
   `.bosun/init.state` so a `bosun init --resume` continues where
   it left off. Today, init failing partway through leaves orphan
   branches + half-populated worktrees that the operator has to
   clean up by hand. We've hit this. Twice.
7. **Agent-crash detection drives session state.** A session whose
   agent process died and whose worktree has uncommitted changes
   gets a new **CRASHED** state. `bosun status` surfaces it; the
   operator runs `bosun rescue session-N` which either: snapshots
   the dirty work to `.bosun/rescues/session-N-<timestamp>/` for
   manual inspection, OR pops a launcher window with the
   half-finished state for the operator to take over.
8. **The "external repo" safety contract.** Document in the README:
   what bosun CAN do to a repo without operator intervention
   (create branches, create worktrees in sibling dirs), what it
   does only on explicit operator command (`merge`, `remove`,
   `cleanup`), and what it never does (touch main without an
   explicit merge command, write outside `<repo>` or `<repo>-bosun-*`).
   Make the boundaries explicit so a user pointing bosun at their
   employer's repo can read the contract and decide.

### Why this is v0.6, not v0.7

The real-repo trial (queued for after v0.5 ships) is the moment
these failure modes go from theoretical to operational. The trial
will produce a punchlist of "the agent in session-3 hit an API 500
and bosun didn't notice for 20 minutes" — and that punchlist becomes
v0.6's scope. Better to land the resilience anchor BEFORE bosun is
public than to discover the gaps when a stranger does.

### Out of scope (still parked)

- **Full transaction-log replay.** Don't store enough state to
  reconstruct an arbitrary session's history; just enough to
  recover from a crash.
- **Sandbox isolation beyond git worktrees.** Don't try to
  containerize agent processes. The operator's git worktree model
  is the boundary; if an agent runs `rm -rf /` it's their problem.
- **Distributed bosun across machines.** Single-machine assumption
  stays through v0.6+. Multi-machine is a v1.0+ research project.
