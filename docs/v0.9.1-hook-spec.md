# v0.9.1 hook spec — PreToolUse auto-claim

Design document for the v0.9.1 work item that closes the
"advisory-claims model is only as good as agent cooperation" gap
flagged in the AI-engineer review of v0.9.0.

The point of writing the spec before any code: the integration
surface is Claude Code's hook contract, which lives outside bosun's
codebase. Settling the design (where the config lives, what runs,
how failures degrade) up front avoids a second redesign once we
discover the hook contract has constraints we didn't model.

---

## What the hook does

When an agent inside a bosun session worktree runs `Edit`, `Write`,
`MultiEdit`, or `NotebookEdit`, a PreToolUse hook intercepts the
tool call and emits `bosun_claim <session> <path>` automatically.
The operator wires the hook once via `bosun hook install`; the
agent never has to remember to call `bosun_claim` again.

Outcome: `bosun status --with-overlaps` becomes reliable instead of
cooperative. The predict/overlap surface stops being load-bearing
on agent convention.

---

## Claude Code's PreToolUse contract

Claude Code fires PreToolUse hooks before any tool call. Each hook
is a shell command that receives the tool invocation as JSON on
stdin and can:

- Exit 0 → the tool call proceeds.
- Exit 2 → the tool call is blocked; stderr is shown to the agent.
- Any other non-zero → block + show the operator a diagnostic.

The JSON shape (relevant fields):

```json
{
  "session_id": "claude-session-uuid",
  "transcript_path": "...",
  "cwd": "/abs/path/agent/cwd",
  "hook_event_name": "PreToolUse",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "/abs/path/to/edit.go",
    "old_string": "...",
    "new_string": "..."
  }
}
```

Per tool, the path field varies:

- `Edit` / `Write` / `MultiEdit` → `tool_input.file_path`
- `NotebookEdit` → `tool_input.notebook_path`

Configuration lives in a JSON file in one of three scopes:

- `<repo>/.claude/settings.json` (project; committed to git or not)
- `<repo>/.claude/settings.local.json` (project, gitignored)
- `~/.claude/settings.json` (user; fires for every claude session)

A PreToolUse entry looks like:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write|MultiEdit|NotebookEdit",
        "hooks": [
          { "type": "command", "command": "bosun hook claim", "timeout": 5000 }
        ]
      }
    ]
  }
}
```

The matcher is a regex against `tool_name`.

## Bosun-side surface

Three new subcommands under a top-level `bosun hook` group:

### `bosun hook install`

Idempotent. Writes the PreToolUse matcher block above into
`<repo>/.claude/settings.json` if absent. Preserves any existing
hooks (appends to the PreToolUse array; doesn't overwrite).

Output:

```
bosun: hook installed at <repo>/.claude/settings.json
       Edit/Write/MultiEdit/NotebookEdit will now auto-claim paths via bosun_claim.
       To uninstall: bosun hook uninstall
```

If already installed:

```
bosun: hook is already installed at <repo>/.claude/settings.json (no changes).
```

### `bosun hook uninstall`

Removes any PreToolUse entry whose command starts with
`bosun hook claim`. Leaves other hooks intact. Removes the
PreToolUse key entirely if it's the only entry; removes the
`hooks` key if PreToolUse was the only nested key.

### `bosun hook claim`

The actual handler. Reads PreToolUse JSON from stdin and:

1. Parse the JSON. On error: log to stderr, exit 0.
2. Extract `cwd` and the relevant path field per `tool_name`.
3. Walk up from `cwd` to find a `.bosun/` parent — that's the
   bosun repo root. If no `.bosun/` found within the worktree
   tree, the agent is editing outside bosun's purview; exit 0.
4. Resolve the session label from the worktree path. The label is
   the suffix of `<repo>-bosun-<label>` derived from the worktree's
   sibling-of-repo path. Helper:
   `session.LabelFromWorktreePath(repoRoot, worktreePath)`.
5. Canonicalize the edited path: make it relative to `repoRoot` so
   the claim matches paths claimed elsewhere through the CLI.
6. Emit `bosun_claim` via either the live MCP socket (preferred,
   sub-millisecond) or by writing directly to `.bosun/claims/`
   through `internal/claims` (fallback when no daemon is up).

The hook NEVER exits non-zero. Bosun errors degrade silently
(stderr only) — the agent's edit must proceed even if the claim
fails. Auto-claim is a convenience; blocking the agent because the
claim file system is misbehaving would erode the trust premise.

## Configuration model: per-repo

`bosun hook install` writes to `<repo>/.claude/settings.json`
(not `settings.local.json`, not `~/.claude/settings.json`).

Rationale:

- **Per-repo** travels with the project. Cloning the repo carries
  the bosun integration with it; new contributors get auto-claim
  for free once they `bosun init`.
- **Per-user** would fire the hook on every claude session on the
  machine — including sessions that have nothing to do with bosun.
  Wrong default.
- **`settings.local.json`** is gitignored and per-developer. The
  hook IS bosun integration; committing it to the repo makes the
  integration discoverable to anyone reading the source. Worth
  the commit.

If `.claude/settings.json` is gitignored by the operator's repo
convention (common), `bosun hook install` still writes there — the
hook just doesn't travel with clones. That's the operator's
choice, not bosun's. Document it.

The install command warns when the path is gitignored:

```
bosun: hook installed at <repo>/.claude/settings.json
bosun: note — .claude/settings.json appears gitignored.
       The hook applies on this machine but won't travel with the repo.
       To make it portable, remove .claude/settings.json from .gitignore.
```

## Path resolution: worktree → session label

`session.LabelFromWorktreePath` (new helper) walks the worktree
path's basename, matches against the `<repo>-bosun-<label>`
pattern from `cfg.WorktreeSuffixPattern`, and returns the label
or empty string.

Edge cases:

- Worktree at `<repo>/` itself (not a session worktree) → empty
  label, exit 0. The hook is no-op for edits in the main worktree.
- Worktree name doesn't match the configured pattern → empty
  label. Maybe the operator hand-created a sibling dir; either
  way, bosun won't claim for it.
- Edited path is outside the worktree (e.g. agent edits a config
  file in `~/.config/...`) → exit 0, don't claim.

Failure modes are silent because the alternative is "your agent
just had its Edit blocked because your worktree dir has an
unexpected name." That's a bug we can't fix from the hook side.

## Error handling: never block

Per the spec above, `bosun hook claim` always exits 0. The full
error matrix:

| Failure | Behavior |
|---|---|
| stdin JSON parse fails | stderr "bosun: hook: parse error: ..." → exit 0 |
| tool_name unknown / not in matcher | exit 0 (we shouldn't have been invoked) |
| Edited path field missing | stderr "bosun: hook: no path in tool_input" → exit 0 |
| Not inside a bosun worktree | exit 0 silently |
| Session label resolution fails | stderr "bosun: hook: couldn't resolve session label" → exit 0 |
| MCP socket present but daemon dead | fall through to direct claim write |
| Direct claim write fails | stderr "bosun: hook: claim write failed: ..." → exit 0 |
| Hook timeout (5s) | Claude Code kills the hook; agent's edit proceeds |

The pattern: bosun's auto-claim is a convenience layer over a
correctness layer (the agent edit). Convenience never blocks
correctness.

## Performance

Claude Code fires PreToolUse before every matching tool call. If
the hook takes 100ms per Edit and an agent does 50 edits in a
session, that's 5 seconds of accumulated overhead — noticeable.

Mitigations:

1. **Fast path: stat the .bosun/ ancestor.** Walking up from
   `cwd` until finding `.bosun/` is one `os.Stat` per ancestor.
   On a typical worktree depth (4-6 levels) this is sub-millisecond.
   If no `.bosun/` found within N levels: exit 0 fast.

2. **MCP socket when available.** If `<repo>/.bosun/mcp.sock`
   exists and accepts a connection, send `bosun_claim` over the
   socket — single round-trip, no fork. Falls back to direct
   write when the daemon isn't running.

3. **Path deduplication.** A burst of edits to the same file
   shouldn't write the claim file 10 times. Cache claimed paths
   per Claude Code session_id (in `/tmp/bosun-hook-cache-<sid>`)
   and skip writes for already-claimed paths.

Target: < 20ms per invocation in the common case (claim already
exists or socket roundtrip).

Pre-build the perf model; profile after a real bosun-on-bosun
session that exercises the hook for an hour. Optimize if measured
overhead > 1% of wall-clock.

## Install idempotency

`bosun hook install` must be safe to run multiple times. The logic:

```python
# Pseudocode for install
settings = read_or_empty(".claude/settings.json")
preToolUse = settings.setdefault("hooks", {}).setdefault("PreToolUse", [])

for entry in preToolUse:
    if any(h.get("command", "").startswith("bosun hook claim")
           for h in entry.get("hooks", [])):
        print("hook already installed; no changes")
        return

preToolUse.append({
    "matcher": "Edit|Write|MultiEdit|NotebookEdit",
    "hooks": [{"type": "command", "command": "bosun hook claim", "timeout": 5000}]
})
write_atomically(".claude/settings.json", settings)
```

Concrete safeguards:

- Atomic write: temp file + rename, same pattern claims.go uses.
- Preserve unknown keys in `settings.json`. The operator may have
  set `permissions`, `model`, etc.; the hook installer only
  touches `hooks.PreToolUse`.
- Preserve unknown keys in the existing PreToolUse entries.
  Append; don't merge.

Uninstall is the inverse with the same atomicity guarantees.

## What's NOT in v0.9.1

- **PostToolUse to release claims on tool failure.** Tempting but
  YAGNI: claims are advisory, releasing them on rare failures
  isn't worth the complexity.
- **Path normalization beyond `filepath.Clean`.** The predict
  heuristic does fancier matching (extension whitelist, etc.);
  auto-claim is for exact paths the agent is actually editing.
- **Notebook-cell-level claims.** The hook claims the `.ipynb`
  file, not the specific cell. Cell-level is over-engineering for
  the auto-claim use case.
- **Auto-suggest hook.** A hook that calls `bosun suggest` on
  every UserPromptSubmit would burn API tokens. Out of scope.
- **Hooks for `Bash` tool calls that mutate files** (`rm`, `sed
  -i`, `git checkout`). The bash matcher would fire on every
  bash invocation, most of which are read-only. Worth revisiting
  in v0.10 if the silent-write case bites real users; for now,
  Edit/Write/MultiEdit/NotebookEdit cover the 95%.
- **Cross-platform hook install.** Claude Code's settings.json
  location matches across macOS/Linux/Windows but the path
  conventions diverge. v0.9.1 targets macOS + Linux paths; v0.9.x
  follow-up adds Windows if a real user needs it.

## Tests

Unit tests in `internal/claudehook/`:

- `hook_test.go` — feeds sample PreToolUse JSON to the handler;
  asserts the expected claim landed for each tool name
  (Edit/Write/MultiEdit/NotebookEdit) and the right path.
- `install_test.go` — install on empty settings, install on
  existing settings with unrelated hooks, install idempotency
  (run twice → one entry), uninstall removes only the bosun
  entry.
- `resolve_test.go` — `LabelFromWorktreePath` against a few path
  shapes (in-worktree, in-main-worktree, outside-any-bosun-tree).

Scenario test in `cmd/bosun/`:

- Real bosun session + real `.claude/settings.json` (using
  `t.TempDir()` for HOME or per-repo). Drive an Edit-shaped tool
  call by piping JSON to `bosun hook claim` via subprocess.
  Verify `.bosun/claims/<session>.json` was updated.

Manual smoke test for the trial:

- `bosun init 1` + `bosun launch session-1` + `bosun hook install`.
- Watch a real Claude Code session in the worktree do a few edits.
- Confirm `bosun status --with-overlaps` shows the claimed paths
  without the agent ever calling `bosun_claim` explicitly.

## Open questions

1. **Hook propagation to sub-sessions.** When `bosun_spawn` creates
   sub-session worktrees, do they inherit the parent repo's
   `.claude/settings.json`? Claude Code's settings precedence walks
   upward from CWD, so a sub-worktree at
   `<repo>-bosun-session-1.auth/` would NOT see the repo's
   `.claude/settings.json` directly (the sibling-not-child path).
   Two options: (a) `bosun_spawn` copies the hook config into the
   sub-worktree's `.claude/` at spawn time; (b) document that
   operators should symlink or commit `.claude/settings.json` into
   the repo so worktrees inherit via the upward walk from
   `<repo>-bosun-*/some-subdir/`. Lean toward (a) — bosun owns the
   spawn, so bosun owns the config carry-over.

2. **Claude Code hook config version.** The PreToolUse contract
   shape may evolve. Bosun should be defensive against unexpected
   shapes — if the JSON has a new field bosun doesn't know about,
   ignore it; if a field bosun expects is missing, fall back to
   silent exit-0. Don't crash.

3. **Bypass via comment annotation.** Operators may want a way to
   say "this Edit is intentional; don't claim." Out of scope for
   v0.9.1 — adds significant complexity (parsing the agent's
   tool-call context) and isn't asked for yet. Revisit if a real
   user needs it.

4. **Hook log location.** `bosun hook claim`'s stderr lines go
   somewhere (Claude Code captures them as hook output). Should
   bosun ALSO log to `.bosun/hook.log` for operator-side
   visibility? Worth doing — operators want to know when the hook
   silently failed. Append-only, rotated by size if it grows.

5. **What about `Read` tool calls?** Read doesn't mutate, so it
   doesn't strictly need a claim — but a session reading a file
   another session is editing is a legitimate conflict shape.
   Defer: claim-on-read would inflate the claim graph for no
   safety benefit. Read sessions aren't the problem; concurrent
   writes are.
