# Bosun v0.5 — Roadmap

**Baseline:** v0.4.0-rc1 (shipped 2026-05-16)
**Target shape:** Finish the lifecycle-hooks story v0.4 scaffolded, harden the kickoff path against the failures `docs/v0.4-findings.md` captured, and add the first piece of "predict where sessions will collide before they actually edit." All small enough that v0.5 can ship as a single round.
**Status:** Planning. Nothing implemented yet; this doc is the agreement before work starts.

---

## The v0.5 anchor

### Finish lifecycle hooks call-sites

v0.4 landed the `internal/hooks` runner and three events (`pre-init`, `post-init`, `post-done`). The remaining call-sites are the v0.5 ask:

- **`pre-merge`** — fires per session, with `BOSUN_SESSION` / `BOSUN_TARGET_BRANCH` / `BOSUN_AHEAD` in the env. A non-zero exit (with `fail_open: false`) blocks the merge — letting an operator wire `make check` or a custom safety gate without modifying bosun itself.
- **`post-merge`** — fires per session after a successful squash. Useful for notifications (Slack post, desktop alert) or for cleaning up adjacent state.
- **`pre-cleanup`** / **`post-cleanup`** — fires around `bosun cleanup` and `bosun cleanup --orphans`. `pre-cleanup` can refuse if e.g. the worktree has uncommitted work the operator wants captured first.
- **`pre-remove`** — fires before `bosun remove`. Same shape as `pre-cleanup` but per-session.

**Why:** the hooks machinery without these call-sites is just plumbing. v0.4 shipped what was safe to scaffold; v0.5 makes it useful.

**Expected change shape:** small. Each call-site is a `hooks.Run(ctx, cfg.Hooks, "<event>", env)` line at the right place in `cmd/bosun/cmd_*.go`. Test additions per event using the existing `internal/hooks` table-driven harness.

## Smaller v0.5 work

### Silent-init-hang fix (`docs/v0.4-findings.md`, Bug 1)

The v0.4 kickoff burned an afternoon to a `git worktree add` that hung silently under APFS fsync pressure. Three independent fixes, all small:

- **Per-operation timeouts.** Wrap every `internal/git.Client.run` call in a `context.WithTimeout`. Default 30s, configurable via `.bosun/config.json` (`git_op_timeout_seconds`). On timeout, return a structured error pointing the operator at filesystem pressure rather than printing nothing.
- **Init progress reporting.** `cmd_init.go` prints "Creating worktree session-N (K/N)..." before each `AddWorktree` call. Operators see forward progress vs. silent hang.
- **Pre-flight load check.** Read `uptime` / `top -l 1` on macOS and `loadavg` on Linux. If 1-minute load > 5, warn but don't refuse. The operator can `--no-load-check` if they want to override.

### Phantom-branch detection (`docs/v0.4-findings.md`, Bug 2)

macOS Finder / Time Machine / Spotlight sometimes duplicates `.git/refs/heads/bosun/session-N` to `.git/refs/heads/bosun/session-N 2`. These show up as warnings on every `git branch` and double-count sessions in overlap reports.

- `bosun init` scans `.git/refs/heads/<session_prefix>/` at startup, detects the `<branch> <digit>` pattern, and offers to delete them with a one-line operator confirmation. `--no-clean-phantoms` opts out.
- README gains a one-paragraph mention of the failure mode + manual cleanup recipe for operators who hit it outside `bosun init`.

### Predictive conflict analysis (first slice)

Today, `bosun claim` is advisory and the operator finds out about collisions when `bosun status --with-overlaps` reports them — i.e., *after* a session has already started touching the path. The v0.5 slice:

- New tool call (and CLI subcommand) `bosun_predict(plan_markdown)` — parse a plan markdown, infer likely paths per session from the brief text using simple heuristics (filename mentions, code block fences, common patterns like `internal/<pkg>/`), and report overlaps before any session opens.
- Heuristic only — no static analysis or LSP integration yet. Goal is "catch the obvious ones," not "guarantee correctness." A miss is fine; a false positive should be rare.
- The full predictive-conflict story (LSP integration, static call-graph analysis) stays parked beyond v0.5.

### Ghostty tab support (pending upstream)

Dropped in v0.2 polish because the Ghostty CLI doesn't support `+new-tab` argv. Track upstream — if Ghostty ships a CLI tab-open mechanism between now and v0.5 cut, wire it in. Otherwise re-park.

### `bosun config` editing surface (round-out)

v0.4 added `bosun config show` / `bosun config set`. v0.5 fills in:

- `bosun config unset <key>` — remove a key, falling back to the default.
- `bosun config validate` — read-only check that the current `.bosun/config.json` parses and that every hook event name is known. Useful as a pre-commit hook or CI gate.
- `bosun config init` — write a stub `.bosun/config.json` with comments showing every supported key and its default. Replaces the current "read the source for available keys" footgun.

## Not in v0.5 (deferred to v0.6+)

- **Full LSP-driven conflict analysis.** v0.5's heuristic prediction is the entry point; the static-analysis version (build a call graph, identify symbol-level edits) needs language-specific work and is too big for one round.
- **Resource governance.** Scheduling test runs across sessions to avoid CPU thrashing, sharing module download caches read-only, etc. Mentioned since v0.2-deferred; still parked.
- **Auth on `bosun serve`.** Today `bosun serve` only binds to loopback by default. A real auth story (operator token / TLS / OAuth) is a v0.6+ feature.
- **Native Ghostty `+new-tab`.** If upstream doesn't ship by v0.5 cut, this slips again.
- **Plugin system / extension points** beyond hooks. Hooks plus the MCP tool surface together cover most of the obvious use cases; we'll only add a richer plugin shape if a real ask surfaces.

---

## Sequence

Suggested round shape (similar to v0.4's round split):

1. **Round 1 — hooks call-sites.** All five remaining events plus tests. Bounded, well-isolated work; one session per pair of events would parallelize cleanly.
2. **Round 2 — kickoff robustness.** Per-op timeouts, progress reporting, pre-flight load check, phantom-branch detection. All in `internal/git/` and `cmd/bosun/cmd_init.go`.
3. **Round 3 — predictive conflict + config round-out.** Independent enough to parallelize.

Each round is a dogfood target; v0.4 proved that the parallel-session lanes work and v0.5 should keep the same shape rather than reinvent it.

## Open questions

1. **`pre-merge` hook semantics under `--dry-run`.** Does `pre-merge` fire during `bosun merge --dry-run`? Argument for: an operator's `make check` hook is exactly what they want a dry-run to exercise. Argument against: dry-run should be side-effect-free, and the hook may have side effects. Leaning toward "don't fire on dry-run; document it."
2. **`pre-cleanup` blast radius.** When `bosun cleanup` reaps multiple sessions, does `pre-cleanup` fire once with a session list, or once per session? Per-session is simpler; once-per-batch is what an operator probably wants for "send one Slack message, not five."
3. **Timeout default.** 30s for `git worktree add` is a guess. Anyone with a working corpus of timing data should set the default empirically before v0.5 cut.
4. **Predictive conflict tool naming.** `bosun_predict` is fine but `bosun_plan_check` or `bosun_dry_claim` may be more discoverable. TBD before round 3 kicks off.
