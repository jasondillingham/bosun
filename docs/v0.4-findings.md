# v0.4 round-1 findings (kickoff failures)

Field notes from attempting to launch the v0.4 round on 2026-05-16.
Not blockers for v0.4 *features* — these are bosun-the-tool gaps
surfaced by the kickoff itself. File for a future round to fix.

## Bug 1: `bosun init` hangs silently when `git worktree add` stalls

**Symptom.** `bosun init 10 --brief …` produced exactly one line of
output ("Created 0 session(s)…" — actually nothing) and stayed silent
for 5+ minutes. `ps` showed `git worktree add` running with 0.02s
CPU time. The on-disk worktree directory had files in it; git was
hung on something post-checkout (most likely an fsync).

**Reproducer (kind of).** Run the macOS system under heavy background
load — bulk delete of caches that triggers Spotlight reindex +
iCloud verification — then run `bosun init N`. Even N=4 hung on the
first `git worktree add`. Load average was 11+; cloudd burning
35% CPU; WindowServer at 44%; 6 stuck processes per `top`.

**Root cause (not bosun's fault, but bosun handles it badly).** APFS
fsync stalls behind whatever else the kernel is flushing. git's
worktree-add finishes the file writes but blocks on the closing
sync. There's no timeout in `internal/git/git.go` for any subprocess
invocation. bosun also prints nothing between "starting init" and
"created N sessions," so the operator can't tell hang from progress.

**What bosun should do.**

1. **Per-operation timeout** on `c.run(ctx, dir, "worktree", "add", …)`
   et al. Default 30s, configurable via `.bosun/config.json`. A
   timeout returns "git worktree add timed out after 30s — likely
   filesystem pressure; check `top` / `iostat`" rather than hanging.
2. **Progress reporting** in `cmd_init.go`. Print "Creating worktree
   session-1 (1/N)…" before each `AddWorktree` call. Operator sees
   forward progress vs. silent hang.
3. **Pre-flight system-load check.** `top -l 1` or `uptime` parsing
   to detect load > 5 (or whatever heuristic). Warn the operator
   "system load is X; init may be slow" but don't refuse.
4. **Recovery on partial init.** If init dies after creating
   worktrees 1..K of N, the operator is left with orphan branches +
   half-populated worktrees that the next `bosun init` won't reuse.
   `bosun init --resume` would re-derive what's done and continue.
   Alternatively, init could be transactional: if step X fails,
   roll back steps 1..X-1.

## Bug 2: macOS Finder phantom branches (`bosun/session-N 2`)

**Symptom.** Repeated runs leave broken-name refs in `.git/refs/heads/`:
`bosun/session-2 2`, `bosun/session-3 3`, etc. These show up as
warnings on every `git branch` and pollute `bosun status
--with-overlaps` (counted as separate sessions).

**Suspected cause.** Some macOS process (Finder? Time Machine?
Spotlight?) duplicates files in `.git/` when it encounters them
during a sweep. The "X 2" suffix is the macOS convention for
duplicate filenames.

**What bosun should do.**

1. **Detect + clean.** `bosun init` could check `.git/refs/heads/` for
   the `bosun/session-N <digit>` pattern and offer to delete them at
   start, with a `--no-clean-phantoms` opt-out.
2. **Document.** Add a `.git/refs/heads/bosun/.gitkeep`-style note in
   the README about the phantom-branch failure mode and the manual
   cleanup (`rm '.git/refs/heads/bosun/session-N N'`).
3. **Investigate root cause.** Whatever is duplicating these files is
   doing it across `~/Documents/Homelab/`. Could be Time Machine's
   Local Snapshots, Spotlight, or a finder operation. Worth one
   reproducible session to nail down.

## Bug 3: Read-only GOMODCACHE files survive `bosun remove --force`

**Already partially fixed** in v0.3.1 by session-3's chmod-before-remove
helper (`internal/git.chmodWritableTree`). But the production
`bosun-bosun-1` cleanup STILL hit Permission denied even with that
fix in place. The realistic-GOMODCACHE-shape test we wrote passes,
which means the test doesn't capture the failure mode.

**What's still needed.**

1. **Capture the actual failing tree.** When this hits again, copy
   the worktree to a side dir BEFORE the rm, then write a test that
   replays it exactly.
2. **Iterative chmod-then-retry-remove loop.** If `git worktree
   remove` fails with EACCES on a specific path, chmod just that
   subtree and retry. Bounded retries (3?).
3. **Investigate macOS file flags.** `chflags noschg` may be needed
   on top of `chmod u+w` for some Go cache contents. Worth checking
   `ls -lO` on a broken tree.

## What we learned for future kickoffs

- Run `df -h ~` and `uptime` before any 4+-session kickoff. Free
  disk < 20G or load avg > 5 means wait.
- Avoid running bulk filesystem operations (cache cleanup, large
  deletes) immediately before a bosun init. Give APFS + Spotlight
  time to drain.
- `--launch` adds N Ghostty windows to the load picture. Skip it
  when the OS is already under pressure; use `bosun launch <session>`
  manually for the few you want to attach to.
