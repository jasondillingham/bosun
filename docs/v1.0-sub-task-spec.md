# v1.0 sub-task spec — `bosun_subtask` MCP tool

Design document for the lightweight, in-context counterpart to
`bosun_spawn`. Tracks issue [#11](https://github.com/anthropics/bosun/issues/11);
adjacent context in [#13](https://github.com/anthropics/bosun/issues/13)
(reframing spawn's pitch) and [#14](https://github.com/anthropics/bosun/issues/14)
(spawn audit log, which sub-tasks should join).

The point of writing this spec before code: `bosun_spawn` and
`bosun_subtask` look similar from a one-line API description but
diverge sharply on lifecycle, claims, hook firing, and observability.
Getting those wrong forces a v1.1 cleanup; getting them right here
is one doc.

---

## 1. Motivation

Trial #3b is the load-bearing data point. The agent was given an
explicit "Use it" brief for `bosun_spawn` and completed three
`Health()` implementations solo in ~30 minutes, never spawning.
From `docs/v0.9-trial-3b-findings.md`:

> "This is high-signal product feedback: a brief that says 'use
> this tool' is not enough to overcome **an agent's preference for
> solo work when the work feels tractable to it.**"

The diagnosis isn't that the agent was wrong. It was right.
`bosun_spawn` writes per-sub briefs, creates worktrees and branches,
launches sub-agents, waits for them to commit, and then merges
their work back. For three serial `client.go` edits that an agent
can complete in 30 minutes, the spawn overhead exceeds the
parallelism win. The agent's calculus — "solo is cheaper" — was
arithmetically correct.

Two product responses follow from that:

- **Issue #13 — reframe spawn's pitch.** Don't sell `bosun_spawn`
  as parallelism. Sell it as context isolation: separate session
  histories, separate claim graphs, mergeable units of committed
  work. Parallelism is a side effect.
- **Issue #11 — make subs cheap.** For work where context isolation
  is *not* what the agent needs (e.g. "audit these 5 packages for
  a specific pattern and give me a summary"), give it a fan-out
  primitive that is roughly free. When subs cost milliseconds
  instead of minutes, the solo-vs-fan-out calculus flips in
  exactly the cases trial #3b flagged.

`bosun_subtask` is the second response. It runs an ephemeral
sub-agent inside the *parent's* worktree, with no branch, no merge
cycle, and no on-disk session record beyond an audit-log row.

## 2. `bosun_spawn` vs `bosun_subtask`

| Dimension              | `bosun_spawn` (today)                                 | `bosun_subtask` (proposed)                          |
| ---------------------- | ----------------------------------------------------- | --------------------------------------------------- |
| Worktree               | New worktree per sub                                  | Shared with parent                                  |
| Branch                 | New branch from parent's HEAD                         | None                                                |
| Merge cycle            | Sub commits → parent merges via `bosun_merge`         | None — output returned inline as tool result        |
| Context isolation      | Full — separate Claude session, separate brief        | None — sub-agent is invoked from parent's process   |
| Setup cost             | Seconds (git worktree, branch, brief, launcher)       | Milliseconds (subprocess + prompt)                  |
| Use case               | "Implement feature X across these three packages"     | "Audit these 5 packages for pattern Y, summarize"   |
| Failure blast radius   | Survives parent crash; orphan worktree recoverable    | Dies with parent; partial output lost               |
| Default lifetime       | Minutes–hours                                         | Seconds–minutes (timeout default 10 min)            |
| Quota / depth gates    | `agent_spawn.max_concurrent_sub_sessions`, `max_depth` | `agent_subtask.max_concurrent` (separate dial); inherits parent's spawn depth, does not consume budget |
| Claim graph            | Independent — sub claims under its own label          | Shared — sub claims roll up to parent's label       |
| Hook firing            | Sub runs in its own Claude session; its hooks         | Parent's PreToolUse hook fires for sub's tool calls |
| Status visibility      | First-class row in `bosun status`                     | Counter on parent's row; full detail in `bosun show <session>` |
| Cancellation           | `bosun_remove --force` after the fact                 | `bosun_subtask_cancel <label>` mid-stream            |
| Audit log (issue #14)  | One record per spawn outcome                          | One record per sub-task call (start + end)          |

The two tools share roughly nothing operationally. They should
share naming convention (`bosun_*`) and live in the same MCP server,
and that's it.

## 3. MCP tool surface

**Working name:** `bosun_subtask`. Alternatives considered:

- `bosun_fanout` — accurate but overcommits to the multiplicity
  case; single sub-tasks are valid too.
- `bosun_probe` / `bosun_scout` — flavor-y; loses the parallel to
  Claude Code's existing `Task` subagent vocabulary.
- `bosun_ask` — too vague; agents will misread it as a question
  channel back to the operator.

`bosun_subtask` mirrors the Claude Code `Task` tool's mental model
(an ephemeral subagent with a scoped prompt) and pairs cleanly with
the heavier `bosun_spawn` (full session) the agent already knows.

### `bosun_subtask` — args

```jsonc
{
  // Parent session label, e.g. "session-3". The MCP server can infer
  // this from the calling socket, but accepting it explicitly keeps
  // the tool callable from tests and surfaced in audit log rows.
  "parent": "string",

  // Short human-readable label for telemetry and status rendering.
  // 3–5 words. Required.
  "description": "string",

  // The full prompt the sub-agent receives. Free-form markdown.
  // Required.
  "prompt": "string",

  // When true (default), the sub-agent's tool set is filtered to
  // read-only tools at the MCP boundary: Read, Grep, Glob, Bash
  // (with a read-only allowlist), WebFetch. Edit/Write/MultiEdit
  // calls are rejected before reaching the parent's PreToolUse hook.
  // When false, the sub-agent inherits the parent's tool set and
  // any writes flow through the normal claim/hook path (§4.1, §4.2).
  "read_only": "boolean (default: true)",

  // Hard wall-clock budget. The MCP server cancels the sub-agent
  // when exceeded. Range 30..1800, default 600.
  "timeout_seconds": "integer (default: 600)"
}
```

### `bosun_subtask` — result

```jsonc
{
  // Synthetic label assigned by the server: "<parent>.sub.<seq>".
  // Stable for the duration of the call; reusable for
  // bosun_subtask_cancel and audit-log correlation.
  "label": "string",

  // The sub-agent's final assistant message. Plain text.
  "output": "string",

  // Token accounting, in the same shape Anthropic SDK returns.
  "usage": {
    "input_tokens": "integer",
    "output_tokens": "integer",
    "cache_creation_input_tokens": "integer",
    "cache_read_input_tokens": "integer"
  },

  // Tools the sub-agent invoked, deduplicated, in call order.
  "tools_used": ["string", ...],

  // Paths the sub-agent claimed (read_only=false runs only).
  // Empty array on read_only=true.
  "claimed_paths": ["string", ...],

  "duration_ms": "integer",

  // "completed" | "timeout" | "cancelled" | "error"
  "outcome": "string",

  // Present only when outcome != "completed".
  "error": "string"
}
```

### `bosun_subtask_cancel` — args

```jsonc
{
  "parent": "string",  // calling session label
  "label":  "string"   // label returned by bosun_subtask
}
```

Result: `{ "cancelled": bool, "reason": string }`. The server
sends a cancellation signal to the sub-agent's loop; if it doesn't
self-terminate within 5 seconds the subprocess is hard-killed. The
underlying `bosun_subtask` call returns with `outcome: "cancelled"`.

## 4. Design questions (from issue #11)

### 4.1 Claim graph: share parent's or independent?

**Shared, under parent's label.**

Sub-tasks run inside the parent's worktree. A read-only sub-task
makes no claims at all — it's strictly observation. A `read_only:
false` sub-task that edits files claims those paths *as the
parent*, so the global claim graph treats parent + active sub-tasks
as a single actor.

Rationale: two sub-tasks editing different parts of the same
package have a real concurrency hazard locally (parent's working
tree), but no inter-session hazard (other sessions just see the
parent moving). Rolling claims up to the parent matches the actual
filesystem locking story. It also avoids a confusing situation
where the operator sees three claimants on the same worktree.

Implication: parent agents that want concurrent writing sub-tasks
must serialize them themselves, or accept the same kind of races
they'd get from running their own tools concurrently. We will not
build intra-parent claim arbitration in v1.0.

### 4.2 Hook firing: does the PreToolUse hook fire for sub-task tool calls?

**Yes — and it must.**

Bosun's PreToolUse hook (see `docs/v0.9.1-hook-spec.md`) intercepts
`Edit | Write | MultiEdit | NotebookEdit` and auto-emits
`bosun_claim` so the safety contract holds without the agent
having to remember. The hook is installed in the parent's Claude
Code settings; sub-tasks invoked from the parent's process inherit
that settings file, so the hook fires for the sub-task's
Edit/Write calls the same way it fires for the parent's.

This is the right behavior. Sub-task writes hit the same working
tree the parent is editing; bypassing the hook would mean
unclaimed writes go through, defeating the v0.9.1 contract for
"every edit in a bosun worktree is claimed."

For `read_only: true` sub-tasks, the MCP server rejects
Edit/Write/MultiEdit tool calls at the MCP boundary *before* the
hook would fire. The hook never sees them. This is intentional:
read-only is a load-bearing capability claim, and enforcing it at
the boundary keeps the hook's job ("convert edits to claims")
unchanged.

### 4.3 Quota: separate dial or share `max_concurrent_sub_sessions`?

**Separate dial.**

`agent_spawn.max_concurrent_sub_sessions` (default 3) exists to
bound the cost of full worktree creation: disk, branches, merge
cycles, operator cognitive load. Sub-tasks have none of those
costs. Bounding them at 3 defeats the motivating use case — "audit
these 5 packages in parallel" wants 5, and the right answer to
"audit these 20 packages" is also "yes," limited by Anthropic API
rate limits rather than bosun's quota.

Proposed config additions:

```json
{
  "agent_subtask": {
    "enabled": true,
    "max_concurrent": 8,
    "max_per_session_per_hour": 100
  }
}
```

Notes:

- `enabled: true` by default. The blast radius is small (no
  worktrees, no branches, no operator-visible artifacts) and the
  read-only default makes the surface roughly equivalent to "the
  agent can re-read files faster." Operators who want it off flip
  it off explicitly.
- `max_concurrent: 8` per parent. High enough that 5-way audits
  work; low enough that an agent fork-bombing into thousands of
  sub-tasks doesn't survive past the dial.
- `max_per_session_per_hour: 100` is a runaway-loop guard. A
  parent that spawns 100 sub-tasks in an hour is doing something
  pathological; refuse with a clear error and force the operator
  to bump the dial.
- Sub-tasks **do not** consume `agent_spawn.max_concurrent_sub_sessions`
  budget and do not count against `agent_spawn.max_depth`. They
  inherit the parent's depth (so a depth-3 spawned session can
  still run sub-tasks) but they don't add to it. Sub-tasks cannot
  themselves call `bosun_subtask` or `bosun_spawn` — that's a
  hard rule in v1.0, revisitable later.

### 4.4 Status visibility: appear in `bosun status` at all?

**Not as their own rows. As a counter on the parent's row, plus
detail in `bosun show <session>`.**

`bosun status` is the operator's at-a-glance "who's doing what."
Its columns assume a unit of work that takes minutes to hours and
ends in a mergeable commit. Sub-tasks complete in seconds, don't
commit anything, and don't need operator intervention. Adding 5
"session-3.sub.4" rows for a 10-second audit fan-out is noise.

Concrete proposal:

- Add a `SUB` column to `bosun status` showing currently-running
  sub-task count for each session. Zero is rendered blank.
- `bosun show <session>` gets a "Recent sub-tasks" section: last
  N sub-task calls with description, outcome, duration, tool
  count. Defaults to N=10.
- Long-form tree views (the eventual `bosun_check_tree` from
  session-2's work, whatever shape it lands in) are free to
  include sub-tasks as terminal nodes; this spec does not depend
  on that work landing in v1.0.

### 4.5 Cancellation: how does the parent stop a runaway sub-task?

**Three layers, listed in order of how often they'll be hit.**

1. **Wall-clock timeout** — every `bosun_subtask` carries a
   `timeout_seconds` value (default 600). The MCP server tracks
   the deadline and cancels the sub-agent when it expires.
   `outcome: "timeout"` is returned to the caller. This handles
   the common "agent got stuck in a tool loop" case without
   parent intervention.
2. **Explicit cancel** — `bosun_subtask_cancel(label)` lets the
   parent abort a sub-task it's still waiting on. Because
   `bosun_subtask` is a synchronous tool call from the parent's
   perspective, this is only useful in two scenarios: (a) the
   parent runs multiple sub-tasks concurrently via Claude's
   parallel-tool-call mechanism and decides one is wasted, or
   (b) a second agent (operator-side) issues the cancel via the
   MCP socket. v1.0 ships the tool; (a) is the expected pattern.
3. **Hard kill on parent exit** — if the parent process dies, all
   its sub-tasks die with it. The MCP server tracks parent PIDs
   and reaps orphaned sub-agents on detection. This is the
   reason sub-tasks are not durable across parent restarts.

The cancellation contract for the sub-agent itself: the MCP server
sends `SIGTERM`; if the agent loop doesn't exit within 5 seconds
the server sends `SIGKILL`. Partial output up to the cancel point
is discarded — sub-task results are all-or-nothing.

## 5. Implementation phasing

Three milestones, each round-able to a separate session lane.

### Phase 1 — MCP surface (foundation)

- New tools: `bosun_subtask`, `bosun_subtask_cancel`. Synchronous,
  one sub-task at a time per parent (no concurrency yet).
- Config schema: `agent_subtask.{enabled,max_concurrent,max_per_session_per_hour}`.
  In Phase 1, `max_concurrent` is hard-pinned to 1; the dial lands
  but isn't honored beyond that.
- `read_only` enforced at the MCP boundary via tool whitelist.
  No write path in Phase 1 — `read_only: false` returns
  `{"error": "write-mode sub-tasks not enabled in v1.0.0"}`.
- Audit log integration: every `bosun_subtask` call writes a
  record to `.bosun/audit/subtask.log` in the schema issue #14
  defines for spawn (extend, don't fork).
- Status: no `bosun status` changes yet. Sub-tasks are invisible
  except via the audit log.

Done when an agent can call `bosun_subtask({prompt: "audit
internal/git/ for nil-pointer risks", description: "nil audit"})`
from inside a session and get a coherent text result back.

### Phase 2 — Per-context isolation enforcement

- Concurrency: honor `agent_subtask.max_concurrent` for real.
  Multiple sub-tasks per parent, each tracked by label.
- Write-mode (`read_only: false`) lands. Sub-task Edit/Write
  calls flow through the parent's PreToolUse hook unchanged
  (§4.2); claims are rolled up to the parent's label (§4.1).
- Depth and quota gates: sub-tasks cannot recurse (cannot call
  `bosun_subtask` or `bosun_spawn`); enforce at MCP entry.
- `bosun_subtask_cancel` lands and is tested against both the
  graceful-exit path and the 5-second-SIGKILL fallback.

Done when an agent can run 5 concurrent read-only audits *and*
a single write-mode sub-task that touches the parent's working
tree without breaking the claim contract.

### Phase 3 — Observability

- `bosun status` gains the `SUB` column. Counts derived from the
  in-memory sub-task tracker exposed via MCP read endpoint.
- `bosun show <session>` gains the "Recent sub-tasks" section.
  Backed by the audit log so it survives MCP restarts.
- If `bosun_check_tree` (session-2 work) ships in time, sub-tasks
  render as terminal nodes there. This spec does not block on
  that integration; the audit log is the durable record.
- Telemetry: per-session sub-task counts, average duration, and
  outcome distribution exposed for the operator's own dashboards.

Done when an operator running `bosun status` during a 20-package
fan-out can see "session-3 SUB=12" and drill in via `bosun show
session-3` to the full call list.

## Recommendation

Land `bosun_subtask` as a deliberately separate tool from
`bosun_spawn`, not a "mode" or flag on the existing one. The two
serve fundamentally different intents — `bosun_spawn` is for
committed, mergeable work that warrants context isolation;
`bosun_subtask` is for ephemeral research that fans out and folds
back into the parent's reasoning. Keeping them distinct lets us
reframe `bosun_spawn`'s pitch per issue #13 ("context isolation,
not parallelism") honestly, fills the gap the trial #3b agent
correctly flagged, and shares the audit-log plumbing from issue
#14. Ship Phase 1 first — a single shared-context, read-only
sub-task is most of the value, and watching real usage before
committing to concurrency, write-mode, and cancellation machinery
is exactly the discipline that kept v0.9 spawn scoped.
