# Bosun v0.6.1 — trial blockers

Two parallel sessions fixing the init-path bugs the v0.6 trial
surfaced (per `docs/v0.6-trial-findings.md`). After this lands, the
trial gets a second attempt; if it passes Phase 2 cleanly, v0.1 ships
publicly per `CLAUDE.md`'s release stance.

**This is a point-release, not a feature round.** Scope is exactly
the two findings — no new features, no adjacent cleanup, no
"while we're here." Keep diffs minimal and surgical so the second
trial isolates whether these fixes actually close the bugs.

**Workflow for every session:**

1. Read this brief in full + read the linked finding in
   `docs/v0.6-trial-findings.md`.
2. Reproduce the bug locally before fixing (write the failing test
   first, watch it fail, then write the fix).
3. Run `make check` before `bosun done`. Race-clean required.
4. Findings outside your lane go in the done message, not in code.

## session-1

**Per-op git timeout audit + fix — `internal/git/` (Finding 1).**

The v0.5 per-op git timeout (default 30s per `docs/v0.5-roadmap.md`)
did not fire when `git worktree add` hung for 14 minutes during the
trial. Either the timeout isn't wrapped around `worktree add`, the
default is much higher than documented, or the timeout fires but the
command silently retries.

**Step 1 — audit.** Read `internal/git/git.go` (or wherever
`Client.run` lives). Trace the call path from
`AddWorktree(ctx, ...)` down to `exec.CommandContext`. Confirm
whether the timeout from `.bosun/config.json`'s
`git_op_timeout_seconds` actually wraps the worktree-add call.
Write a one-paragraph audit note in the commit message describing
what you found.

**Step 2 — regression test.** Before any fix, add a test in
`internal/git/git_test.go` that:

- Shims a fake `git` binary onto PATH that sleeps 60 seconds when
  invoked with `worktree add`.
- Calls `client.AddWorktree(ctx, ...)` with a 2-second timeout.
- Asserts the call returns within ~3 seconds with a timeout error
  (not after 60s with a hung-process error).

This test MUST fail against current main. Watch it fail. Then write
the fix.

**Step 3 — fix.** Whatever the audit found:

- If `worktree add` isn't timeout-wrapped, wrap it.
- If the default is too high (or zero-means-infinite), set a
  sensible default (60-120s — worktree-add under fsync pressure is
  legitimately slow; 30s was probably too aggressive for that
  specific operation. Other git ops can stay at 30s).
- If the timeout fires but the command retries silently, kill the
  retry — fail fast and let `bosun init --resume` handle recovery.

**Step 4 — error message.** On timeout, the error must say:

```
git worktree add timed out after Xs — system likely under fsync
pressure; check `uptime` and retry, or `bosun init --resume` to
continue.
```

The pointer to `--resume` is critical — once session-2 lands,
that's the recovery path. Without the pointer the operator is
stuck.

**Files (own):**
- `internal/git/git.go` (modify)
- `internal/git/git_test.go` (modify)
- Any other file under `internal/git/` if the timeout wrapping lives
  elsewhere (audit first, then identify owned files).

**Files (avoid):**
- `cmd/bosun/cmd_init.go` — session-2
- `internal/init/*` — session-2
- `internal/predict/*` — not in scope this round

## session-2

**`bosun init --resume` fix — `cmd/bosun/cmd_init.go` + `internal/init/` (Finding 2).**

Two distinct bugs in the resume path, both fatal:

**Bug 2a — argless `bosun init --resume` refuses with arg-count
mismatch:**

```
$ bosun init --resume
Error: bosun: --resume label count (4) doesn't match prior init (3).
```

Where does "4" come from? Audit the flag-parsing path in
`cmd_init.go`. Most likely cause: the default for the positional
session-count is set to something (or `len(args)+1`) before
`--resume` is consulted, so the comparison fires on a phantom
default.

**Fix:** `--resume` should derive everything from `init.state`:
session count, brief path, label set. Operator should be able to
run `bosun init --resume` from anywhere in the repo with zero
other args, and it picks up cleanly. If the operator passes
arguments alongside `--resume`, those become an override (with a
warning) rather than a hard mismatch error.

**Bug 2b — resume refuses on the locked-in-progress worktree it's
supposed to recover:**

```
$ bosun init 3 --resume --brief v0.6-trial-plan.md
Error: bosun: worktree path already exists: ...-bosun-1
       (use --force to overwrite)
```

When `init.state.current_step == "git_worktree_add"` and the
worktree exists + is registered in `git worktree list`, that's the
recoverable case the breadcrumb exists for. Resume must:

1. Read `init.state`. Identify `current_session` and `current_step`.
2. Run `git worktree list --porcelain` — is the in-progress
   worktree registered?
3. If yes:
   - If it's `locked` (from bosun's prior kill), `git worktree
     unlock` it.
   - Mark `git_worktree_add` step complete.
   - Advance to the next step (`state_file_write` → `hook_post_init`
     per the v0.6 schema).
4. If no (worktree dir exists on disk but not registered with git):
   - That's the case `--force` should handle. Refuse without
     `--force`; with `--force`, `rm -rf` the dir and re-create.
5. If yes AND lock owner is some other process (not bosun's kill):
   - Refuse with a clear "another process holds the lock" error.
   - Operator can `git worktree unlock` manually if they know better.

**Bonus — `bosun init --status` (only if time permits):** new
read-only subcommand that reads `.bosun/init.state` and prints what's
done, what's pending, when the prior init started. Lets an operator
inspect a wedged init without taking action. Skip if the resume work
takes the full lane budget — `--status` is nice-to-have, not a
trial blocker.

**Tests** in `cmd/bosun/cmd_init_test.go` and `internal/init/state_test.go`:

- `bosun init --resume` with no other args derives count + brief from
  init.state.
- Resume detects existing-registered-locked worktree, unlocks +
  advances rather than refusing.
- Resume detects existing-unregistered worktree dir, refuses without
  `--force`.
- Resume with `--force` overwrites a stale worktree.
- Inconsistent args (different count, different brief path) emit a
  warning + use init.state values, NOT a fatal error.

Scenarios test in `scenarios_test.go`:

- Start `bosun init 3` against a fake-git that hangs on the 2nd
  worktree-add (use session-1's timeout fix so this terminates).
- After timeout, run `bosun init --resume` and verify all 3 sessions
  finish cleanly.

**Files (own):**
- `cmd/bosun/cmd_init.go` (modify)
- `cmd/bosun/cmd_init_test.go` (modify if exists; else add cases
  in `scenarios_test.go`)
- `internal/init/state.go` (modify — may need a `ResumeFrom()` helper
  that does the worktree-list reconciliation)
- `internal/init/state_test.go` (modify)
- `cmd/bosun/scenarios_test.go` (append at EOF for end-to-end resume

## Out of scope (deferred to v0.6.2 or v0.7)

These came up in the trial findings but aren't blockers for the
second trial attempt. Resist the urge to fold them in.

- **Predictor's "Files (avoid)" false-positives** (v0.6 round-1
  surfaced this). The heuristic counts paths mentioned under
  "Files (avoid)" headings as predicted edits, producing 79
  false-positive overlaps on the v0.6 round-1 plan. Real bug, real
  fix needed, but `bosun predict` is advisory — operator can ignore
  it during the trial. Park for v0.6.2 or v0.7.
- **Default behavior of `bosun launch <session>` without
  `--initial-prompt`** — the launched claude sits at an empty prompt.
  Fixed manually in v0.6 round 1 by re-launching with the prompt;
  the default should match `bosun init --launch`'s behavior. Park.
- **README "Safety contract" section validation against trial
  finding #6.** Lift the trial's "HEAD unchanged through hang +
  recovery" observation into the README's safety contract. Park
  until after the second trial confirms it generalizes.
