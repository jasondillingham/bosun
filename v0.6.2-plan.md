# Bosun v0.6.2 — timeout coverage gap

Single session closing the bug surfaced in the v0.6.1 dogfood round
itself: per-op git timeout exists in `internal/git/Client.run` but
several git invocations bypass it by calling `exec.CommandContext`
directly. Result: `bosun merge` hangs forever on `git fsck` under
load (observed 11 min, 0.22s CPU during v0.6.1 round).

**Scope is exactly this bug.** No new features, no adjacent cleanup.
After this lands, re-run the v0.6 external-repo trial against
`homelab-status-mcp` per `docs/v0.6-trial-protocol.md`.

---

## session-1

**Centralize git timeout enforcement — `internal/git/fsck.go` + `cmd/bosun/cmd_merge.go`.**

The v0.5/v0.6.1 timeout machinery is correct where it's used:
`(*Client).run` wraps `exec.CommandContext` with the configured
timeout, and `AddWorktree` even has a longer per-op floor for the
fsync-heavy worktree-add case. But three call sites bypass the
client entirely and run git directly:

- `internal/git/fsck.go:40` — `FsckWorktree` calls `exec.CommandContext`
  with the caller's context, which has no per-op timeout applied.
- `cmd/bosun/cmd_merge.go:404` — `git rev-parse HEAD` for pre-merge
  SHA capture.
- `cmd/bosun/cmd_merge.go:704` — `git reset --hard <sha>` for
  `merge --undo`.

Each of these can hang indefinitely under filesystem pressure (fsck
is the one we hit; the others are smaller surface but same shape).

**Step 1 — regression test that fails on current main.** In
`internal/git/fsck_test.go`, add a test that:

- Shims a fake `git` binary onto PATH that sleeps 60s for any
  invocation.
- Calls `FsckWorktree(ctx, dir)` with a `Client` configured for a
  2-second timeout.
- Asserts the call returns within ~3 seconds with a timeout-shaped
  error.

The test must fail against current main. Watch it fail before
writing the fix.

**Step 2 — migrate FsckWorktree to the Client.** Convert
`FsckWorktree(ctx, dir)` into `(c *Client).FsckWorktree(ctx, dir)`
so it calls `c.run(ctx, dir, "fsck", "--no-progress",
"--no-dangling")` — picking up the configured timeout
automatically. Update the call site in `cmd_merge.go:250` to use
the client method. Existing `FsckError` shape preserved (the run
helper returns the same exec.ExitError shape; the error-wrapping
logic in fsck.go stays put, just shifts to consume `c.run`'s
output).

**Step 3 — audit the two cmd_merge.go bypasses.** Replace lines
404 (`rev-parse HEAD`) and 704 (`reset --hard <sha>`) with
`Client` calls. `RevParseHEAD(ctx, dir)` and `ResetHard(ctx, dir,
sha)` are good helper names — keep them in `internal/git/git.go`
next to the existing helpers.

**Step 4 — sweep for other bypasses.** `grep -rn
"exec\.CommandContext.*git" --include="*.go"` and verify every
remaining hit is either: (a) test code, or (b) something that
genuinely doesn't go through the Client and needs migrating in
this round. Document any unconverted sites in the commit message
with rationale.

**Step 5 — error message.** When the timeout fires, the wrapped
error from `Client.run` should clearly say which git op timed
out:

```
git fsck timed out after 30s — system likely under fsync pressure;
check `uptime` and retry, or `bosun cleanup` to drop this session
and start fresh.
```

The recovery pointer depends on the op — fsck timeout points at
retry/cleanup; reset-hard timeout points at `git reflog` (the
operator's escape hatch). Both can share the "fsync pressure"
diagnostic since that's the actual cause.

**Step 6 — test coverage at the Client.run level.** In
`internal/git/git_test.go`, add a single table-driven test that
spins up the fake-hung-git shim once and exercises every public
method on `*Client` (CreateBranch, DeleteBranch, AddWorktree,
RemoveWorktree, ListWorktrees, RevListCount, FsckWorktree,
RevParseHEAD, ResetHard). Each row asserts the method returns
within timeout+1s. This is the regression-proof against future
"oh I'll just call exec.CommandContext here" backsliding.
