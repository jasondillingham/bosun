# Bosun `suggest` — Brief-authoring assistant spec

**Status:** Draft. Slated as a co-anchor for v0.5 alongside the
lifecycle-hooks finish (`docs/v0.5-roadmap.md`).

## The problem

Today, using bosun productively requires the operator to author N
disjoint per-session briefs in a markdown plan file. The operator has
to:

1. Know the goal they're trying to achieve across N agents.
2. Break the work into N lanes that don't touch the same files.
3. Write a concrete brief for each lane: scope, owned files, files
   to avoid, notes about gotchas.
4. Optionally declare dependencies between lanes.

Step 2 is the hard part. The lane-overlap merge pain we've hit in
every dogfood round so far is fundamentally a brief-quality issue.
Operators who haven't done this before either don't decompose the
work cleanly, write briefs too narrow (under-utilizing parallelism),
or write them too wide (collision city).

Bosun's adoption ceiling, as of v0.4, is "people who are already good
at decomposing work into disjoint parallel coding tasks." That's a
narrow audience. `bosun suggest` is the move that broadens it.

## Goal

Give bosun a `suggest` subcommand and matching MCP tool that takes a
high-level goal and produces a complete, disjoint-by-construction
plan markdown the operator can edit and feed to `bosun init`. The
operator keeps ownership of the plan; the assistant removes the
blank-page problem.

## Non-goals

- **Automatic kickoff.** `bosun suggest` writes a file, then stops.
  Operator reviews + edits + runs `bosun init --brief …` themselves.
- **Cross-language LSP / call-graph analysis.** That's separate v0.6+
  work. v0.5's `suggest` uses heuristics + Claude reasoning, not
  symbol-level tracking.
- **Replacing the operator's judgment.** Even a perfect lane-design
  pass benefits from a human read before launching the swarm.

## End-user UX

```
$ bosun suggest "Migrate the codebase from React Router v5 to v6" --sessions 6

Inspecting repo (3s)
  - 142 .tsx / 38 .jsx files; 47 use react-router-dom
  - Recent commits suggest in-flight auth refactor (#221, #218)
  - Tests under src/__tests__/, fixtures in src/test-utils/
  - go.mod / package.json: react-router-dom ^5.3.4

Proposing lanes (8s)
  Wrote suggested-plan.md (6 lanes, 0 file-claim overlaps)

  session-1: Introduce v6 compatibility shim (depended on by 2-6)
  session-2 (depends: session-1): Migrate /auth routes
  session-3 (depends: session-1): Migrate /dashboard routes
  session-4 (depends: session-1): Migrate /settings routes
  session-5 (depends: session-1): Migrate route fixtures + tests
  session-6 (depends: session-2, 3, 4): Strip remaining v5 imports

Review suggested-plan.md, edit if needed, then:
  bosun init 6 --brief suggested-plan.md --launch
```

`--sessions N` defaults to `cfg.DefaultSessionCount`. Other flags:

- `--out <path>` — output file (default `suggested-plan.md`)
- `--model <name>` — override the Claude model used. Default
  `claude-sonnet-4-6` (or whatever the configured production model
  is at release time).
- `--inspect-only` — print the gathered repo intel and stop, no
  Claude call. For debugging the inspection step.
- `--max-tokens <n>` — bound the LLM response. Default 8000.

## Architecture: three phases

### Phase 1 — Inspect the repo

Goal: produce a compact, accurate snapshot of the repo's shape +
recent activity that a model can reason about. Cap output at ~6KB
so we don't burn input tokens on noise.

Implemented in new package `internal/suggest/inspect.go`. Reads:

- **Shape:** total file count, top-level directory tree (3 levels
  max), file-extension histogram (top 10), repo language(s) inferred
  from `package.json` / `go.mod` / `Cargo.toml` / `pyproject.toml`
  / etc.
- **Recent activity:** last 30 commit subjects (`git log -30
  --pretty=format:'%s'`). Helps the model notice in-flight work to
  avoid colliding with.
- **Open files in HEAD:** `git ls-files | head -200` for a sample.
  For repos > 200 files, sample randomly (deterministic seed: SHA of
  HEAD) so the same goal produces the same lanes.
- **Test layout:** common patterns (`*_test.go`, `*.test.ts`,
  `tests/`, `__tests__/`) — hint to the assistant where test files
  go so lanes can co-locate their tests.
- **Dependencies:** parsed `package.json` / `go.mod` / etc. — the
  list of installed libraries. Useful when the goal mentions one of
  them.
- **Existing bosun config:** `.bosun/config.json` content (mostly
  for `worktree_suffix_pattern` and `session_prefix`).

Output is a `RepoIntel` struct, JSON-serialized for inclusion in the
prompt.

### Phase 2 — Propose lanes

Goal: a Claude API call that takes (goal, repoIntel, N) and returns
a structured proposal of N lanes — name, scope, owned files (as
globs), avoid files (as globs), depends_on (label list), and a
rationale.

The prompt is the highest-leverage piece. Key elements:

- **System prompt** establishes the role: "You're proposing parallel
  coding lanes for N agents. Each lane must own files no other lane
  touches. The operator has tried this before — you've seen real
  agents collide on shared types when they ignored your own/avoid
  lists. Cycles are forbidden."
- **Few-shot examples** drawn from `v0.3-plan.md` and `v0.4-plan.md`
  — these are gold-standard hand-authored plans. Strip them down to
  the structural shape (one or two of the actual session sections),
  inline them in the user prompt as "here's what a good plan looks
  like."
- **Lane-design rules**, plain-text, told to Claude:
  - Each lane owns files no other lane touches (by glob).
  - Lanes with structural setup work get fewer downstream peers.
  - If foundational changes need to land first, put them in lane 1
    and have downstream lanes declare `depends: session-1`.
  - Cycles are not permitted; bosun will refuse them.
  - Test files belong to whichever lane owns the production code
    they cover.
- **Goal + repoIntel** as user-provided context.
- **Response format** specified as JSON conforming to a small
  schema:

```json
{
  "version": "v1",
  "sessions": [
    {
      "label": "session-1",
      "scope": "one-line summary",
      "owned_files": ["internal/auth/**", "cmd/bosun/cmd_login.go"],
      "avoid_files": ["internal/storage/**"],
      "depends_on": [],
      "rationale": "1-2 sentences for the operator review",
      "work_to_do": [
        "bullet point 1",
        "bullet point 2"
      ],
      "notes": "optional gotchas"
    }
  ]
}
```

JSON output is checked against a schema before continuing. If the
model returns malformed JSON or violates the schema, retry once with
the validation error fed back.

### Phase 3 — Write the plan markdown

Goal: convert the structured lane proposal into the prose
`suggested-plan.md` file the operator + bosun consume.

This is mostly templating, not a second LLM call. Use the existing
brief format (the v0.4-plan.md shape) with substitution:

- Top of file: introduction paragraph naming the goal.
- Per lane: `## session-N` (or `## <label>` if named) heading,
  optional `(depends: …)` clause, then a section with the scope,
  work-to-do bullets, owned/avoid file lists, and notes.
- Bottom of file: a "Sequencing notes" block summarizing the
  dependency graph (mostly for the operator's mental model).

Optionally — and this is where the spec gets opinionated — a *second*
short LLM call can polish the work-to-do bullets into prose that
reads like an experienced engineer wrote them. Default off; behind
`--polish`. Reason: a polish call doubles the latency + cost of
`suggest` and the structural output is usable on its own.

## Cost & latency model

- Phase 1 is local; no API cost. Should run in <2s on a typical repo.
- Phase 2 is one Claude API call. Input ~6KB repoIntel + ~3KB prompt
  scaffolding = ~9K input tokens. Output ~4K tokens at max.
  Estimated cost per call: ~$0.05 at Sonnet 4.6 pricing. Latency
  ~6-10s.
- Phase 3 is local templating; <100ms.
- `--polish` adds a second ~3K input / ~3K output call: ~$0.03 more.

Net: ~$0.05-0.08 per `bosun suggest` invocation. That's roughly the
cost a user perceives as "free."

## Failure modes & their handling

- **No Claude API key.** Print a clear error pointing at the env var
  (`ANTHROPIC_API_KEY`). Skip the network call entirely; don't fall
  back to a "dumb" mode that produces a bad plan.
- **Goal too vague.** If the model returns lanes whose `scope` is
  uniformly "do part of the work," the spec is too broad. Print the
  output and a suggestion to narrow the goal. Don't try to be
  clever; the operator iterates.
- **Model returns overlapping files across lanes.** Validate the JSON
  by computing the union of `owned_files` globs and checking for
  overlap. If any, retry once with a feedback message; if the retry
  also overlaps, write the plan anyway and surface the overlap in
  the output so the operator can hand-edit before kickoff.
- **Model returns a dependency cycle.** Reuse `brief.FindDependencyCycle`
  from v0.3 to detect. Retry once; if still cyclic, refuse and tell
  the operator.
- **Repo too big to summarize in 6KB.** Detected by total file count
  > 5000. Print a warning + suggest narrowing the goal or running
  `bosun suggest --root <subdir>` (future flag, not in v0.5 scope).
- **API timeout / network error.** Retry with exponential backoff
  (3 tries). Fail with a structured error pointing at the network.

## Configuration

`.bosun/config.json` gains:

- `suggest.model` (string, default `"claude-sonnet-4-6"`)
- `suggest.max_tokens` (int, default 8000)
- `suggest.polish` (bool, default false)
- `suggest.api_key_env` (string, default `"ANTHROPIC_API_KEY"`)

All overridable via flags.

## Testing strategy

Three layers:

1. **`internal/suggest/inspect_test.go`** — unit-test the repo
   inspector against `t.TempDir()`-bootstrapped repos with known
   shape. Verifies the JSON shape, the file sampling determinism,
   the language inference heuristics.
2. **`internal/suggest/lane_test.go`** — unit-test the JSON-schema
   validation, overlap detection, cycle detection, retry logic.
   Uses a stubbed Claude client (interface in the package; real
   implementation in `internal/suggest/claude.go`) so tests don't
   need network or API keys.
3. **`cmd/bosun/scenarios_test.go`** — one or two end-to-end
   scenarios driving the CLI with a recorded Claude response (JSON
   fixture in `testdata/`). Verifies the plan markdown comes out
   well-formed and the markdown survives a round-trip through
   `brief.Parse`.

No integration test that hits the real Claude API in CI — too brittle
and expensive. The `claude.go` interface gets one optional
`go test -tags=integration` test the maintainer runs locally before
release.

## Open questions

1. **Should `bosun suggest` write the plan into `.bosun/briefs/`
   immediately, or to the repo root?** Operator-edit-then-init
   suggests root (closer to dogfood-plan.md / v0.4-plan.md
   convention). Auto-archive to `.bosun/briefs/plan.last.md` after
   `bosun init` consumes it.
2. **Does `suggest` consume the operator's plan history?** A v0.6
   future could feed prior plans + their merge outcomes back in
   ("the last 3 times you used this lane shape, sessions 4 and 5
   conflicted on `internal/auth/`"). v0.5 stays stateless.
3. **MCP tool surface.** `bosun_suggest` as an MCP tool means a
   Claude Code session inside a worktree can ask bosun for a fresh
   plan mid-round. Powerful but recursive — a session asking bosun
   to fan out *more* sessions. Defer to v0.6 unless a clear use case
   surfaces.
4. **What if the operator passes a non-English goal?** Probably
   fine — Claude handles it — but worth a smoke test before release.
5. **Does the spec belong in this round, or should `suggest` be its
   own dogfood round?** Probably its own round; it's substantial
   enough that mixing it with hooks call-sites would dilute focus.
   Sequence note in `v0.5-roadmap.md` should reflect this.
