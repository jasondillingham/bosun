# Bosun v0.7 — Roadmap

**Baseline:** v0.6.2 (timeout coverage centralized; trial #2 pending re-run on a healthy machine).
**Anchor:** **Polish.** Five recurring operator-friction items surfaced across the v0.4 → v0.6.2 dogfood loop. None of them violate the safety contract; all of them rough up the experience enough to be worth closing before the next anchor round.
**Status:** Planning. The candidates were captured live in `STATUS.md` during the v0.6.2 trial-prep session and revisited here.

---

## Position relative to the public launch

Per `CLAUDE.md`'s release stance, v0.1 ships publicly **only after** the external-repo trial validates the safety contract. The v0.6.2 trial attempt did not reach Phase 2 (system was under crushing load); it needs a re-run on a healthy machine before the gate is cleared.

**v0.7 does not gate the public launch.** v0.6.2 is the last work that does. v0.7 is queued **after** the public-launch flip — it's the first round where bosun is being used by someone other than the maintainer, and the candidates below are the items most likely to bite a newcomer.

That said, the prep is happening now (this doc + `v0.7-round1-plan.md`) because the trial #2 wait is long enough to be productive. Lanes are ready to fire the moment trial #2 passes and v0.1 is shipped publicly.

---

## Why polish, and why now

Three observations from the v0.4 → v0.6.2 dogfood arc:

1. **The "compounding > commits" principle is working, but the floor is real.** Five separate operator-frictions accumulated across rounds where each round's anchor work (suggest, resilience, timeout) deliberately ignored them. Holding the line on scope was right. Doing it forever isn't — at some point the un-addressed friction degrades the new-user experience faster than the anchor work improves it.
2. **None of the candidates are net-new feature surface.** They're all "the existing surface is rougher than advertised" bugs. Per the v0.6 roadmap's NOT-in list, this is exactly the kind of work that *should* happen between anchor rounds, not bundled into them.
3. **The public-launch readme makes promises this work backs up.** `bosun launch <session>` is in the documented command list; today it opens an empty prompt. `bosun rescue` is documented as the crash-recovery path; today it can't handle the corrupted-gitdir case we hit during v0.6.2 session-1. Better to close the gap between the docs and the behavior before strangers find it.

## The v0.7 asks

Five items, grouped into four disjoint lanes for parallel execution. Numbered by surfaced-order in `STATUS.md`'s v0.7-candidates section (not severity — see the lane breakdown for sequencing).

### 1. `bosun launch <session>` default initial prompt

Today `bosun launch session-N` (standalone, no `--initial-prompt`) opens Claude with no first message — the agent sits idle until the operator types something. Meanwhile `bosun init --launch --brief X` injects "Read BOSUN_BRIEF.md..." automatically. Standalone launch should match: if no explicit `--initial-prompt` is passed AND the session worktree contains `BOSUN_BRIEF.md`, default to the same Read-brief prompt.

**Why this matters:** the standalone launch path is the one operators use to relaunch a session after a Ghostty close, an OS sleep, or a `bosun rescue`. Sitting at an empty prompt is a confusing UX nudge to do nothing — exactly the wrong shape for the relaunch case.

### 2. Predictor false-positives on "Files (avoid)" sections

`bosun predict` treats paths listed under "Files (avoid)" headings in a brief as **predicted edits** for that session. The v0.6 round-1 plan triggered 79 false-positive overlaps from this alone — every "avoid" list across four lanes got counted as four conflicting claims. The predictor's whole purpose is to give the operator a heads-up about real conflicts; drowning the signal in false positives is worse than not having the predictor at all.

**Fix shape:** when scanning a brief's session section, skip lines under "Files (avoid)" / "Files (do not touch)" / "Do not touch" headings. Treat them as exclusions, not predictions.

### 3. `bosun init` and stale session branches

If a session branch (e.g. `bosun/session-1`) already exists from a prior round — typically because a prior `bosun cleanup` deleted the worktree but not the branch — `bosun init` today creates a fresh worktree pointing at the **stale commit** that branch references. The operator sees a worktree that looks fine but contains pre-cleanup code. If they `bosun done` + `bosun merge` without noticing, the squash includes whatever the stale branch had.

**Fix shape:** before `git worktree add`, check if `bosun/session-N` already exists. If yes:
- If it points at the configured base branch HEAD → reset it to the new base and proceed (silent, common case after a clean cleanup that retained the branch).
- If it points at a divergent commit → refuse with a clear error: `bosun/session-N already exists at <SHA> (not <base>). Run 'git branch -D bosun/session-N' to drop the stale branch, or use --force to reset it.`

`--force` opts into the reset path.

### 4. macOS Finder phantom `.done` files

Spotlight / Time Machine / iCloud Drive periodically duplicate files inside `.bosun/state/` — `.bosun/state/session-1 2.done`, `.bosun/state/session-1 2.json` keep appearing. The bosun state reader sees the duplicate as a separate (malformed) session record and surfaces it in `bosun list` / `bosun status` as a phantom session that doesn't have a real worktree.

**Fix shape:** when reading `.bosun/state/*.{done,json}`, skip files whose name matches the macOS Finder duplicate pattern (`<name> N.<ext>` where N is a digit) **and** also drop a `.bosun/.metadata_never_index` marker file as a belt-and-suspenders signal to Spotlight to stop indexing the directory.

### 5. `bosun rescue` detection of corrupted worktree gitdir

Observed during the v0.6.2 round itself: session-1's `.git/worktrees/bosun-bosun-1/` directory was reduced to just `index` (no `HEAD`, no `commondir`, no `config`) after the agent crashed under crushing load. The worktree is then in a state where **all git operations fail** ("fatal: not a git repository") but `bosun rescue` happily proceeds to snapshot it, runs into the broken git, and emits a confusing error.

**Fix shape:** before `bosun rescue` runs, validate the worktree gitdir has the expected layout (`HEAD`, `commondir`, `gitdir`, `index` all present). If corrupted, refuse with a clear error pointing the operator at the right manual recovery:

```
bosun: session-N's worktree gitdir is corrupted (.git/worktrees/<name>/ missing
       HEAD/commondir). The agent crashed before writing the gitdir cleanly.
       Recovery:
       - bosun remove session-N --force  (preserves uncommitted files via rescue snapshot)
       - then re-init the session from scratch.
```

Plus: `bosun remove --force` snapshots any salvageable files to `.bosun/rescues/` before nuking, so the operator doesn't lose work.

## What's explicitly NOT in v0.7

Two STATUS.md candidates are intentionally deferred further:

### Self-experimentation hazard (STATUS.md item #6)

When bosun is working ON bosun, session-N's worktree IS a bosun worktree of the bosun repo. v0.6.1 session-2 ran `git worktree remove` on its own worktree while testing the init-resume code and orphaned the branch + commits. STATUS.md notes this is *"not a real concern for external-repo trials."* Defer until either (a) a public user reports hitting it, or (b) it actually causes data loss during in-house dogfood (so far it's been recoverable).

### Scenarios test file split (STATUS.md item #7)

The recurring conflict on `cmd/bosun/scenarios_test.go` (every lane appends at EOF) is real but doesn't need a refactor. The v0.7 round establishes a **new convention**: each lane creates `cmd/bosun/scenarios_<feature>_test.go` for its own scenarios, leaving the legacy `scenarios_test.go` for pre-v0.7 scenarios. No churny rename, no migration; the convention spreads naturally as new lanes pick up the pattern.

This is documented in the v0.7 round-1 brief itself, not as a separate ask.

## Sequence

Four disjoint lanes, no cross-file dependencies:

1. **Lane 1 — Launch UX (ask #1).** `cmd/bosun/cmd_launch.go` + its test file. Smallest lane; effectively one if-statement plus a regression test.
2. **Lane 2 — Predictor accuracy (ask #2).** `internal/predict/predict.go` + tests. Self-contained to the predictor package.
3. **Lane 3 — Pre-flight robustness (ask #3 + the v0.6.2-parked merge pre-flight).** `cmd/bosun/cmd_init.go`, `cmd/bosun/cmd_merge.go`, `internal/init/state.go`, plus extracting `getLoadAverage` into a new tiny `internal/preflight/` package so both init and merge can consume it.
4. **Lane 4 — State + rescue resilience (asks #4 + #5).** `internal/state/state.go`, `cmd/bosun/cmd_rescue.go`, plus a new `internal/git/worktree_health.go` helper for the gitdir validation.

Each lane creates its own `cmd/bosun/scenarios_<lane>_test.go` to avoid the EOF-conflict pattern (see deferred-further section).

## After v0.7

- **Re-trial against an external repo** if any of the lanes touched behavior that's load-sensitive (the rescue/state lanes especially). Use the same `homelab-status-mcp` repo as the v0.6 trial baseline.
- **v0.2 public release** if v0.7 surfaces nothing new — bump the README's compatibility section and tag.
- **Defer v0.8 planning** until v0.7 ships and the public has had at least a week of contact with the released v0.1.

Resist the urge to bundle v0.8 candidates into v0.7. The compounding-not-commits principle still applies; v0.7 is the polish round, not a feature-add round.

## Open questions

1. **Should v0.7 land before the public launch as v0.6.3?** Argument for: asks #3 and #5 are arguably safety-adjacent (stale-branch merge of wrong code; rescue can't actually rescue). Argument against: both require unusual preconditions (cleanup that retained branches; agent crash under crushing load), the user has the safety-contract observation that *main is never touched without explicit merge*, and the public-launch can ship with these as known edge cases documented in the README. Current call: stay v0.7 (post-launch), revisit if trial #2 surfaces either failure mode under more reasonable load.
2. **Default-prompt phrasing for ask #1.** Match init's exact string ("Read BOSUN_BRIEF.md and follow the instructions for your assigned session"), or use a launch-specific phrasing that hints at the re-launch case ("Resume work on the assigned session. Read BOSUN_BRIEF.md if you need context.")? Lane operator's call — bias toward init-symmetry unless there's a strong reason.
3. **Finder duplicate detection threshold for ask #4.** Strict regex (`* 2.done` only) catches the common Spotlight case but misses iCloud Drive's `*.icloud` and `* (1).*` shapes. Permissive regex catches more but risks ignoring a legitimately-named state file. Lane operator's call — bias strict unless the trial surfaces broader shapes.
