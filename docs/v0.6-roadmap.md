# Bosun v0.6 — Roadmap

**Baseline:** v0.5 (suggest + hooks call-sites + kickoff robustness + predictive conflict, shipped 2026-05-15)
**Anchor:** **Resilience.** Everything in v0.6 exists to answer one question — *what happens when an agent dies, the API drops a 500, the operator's machine reboots, or a hook hangs?* — well enough that bosun can point at someone else's repo without risking their code.
**Status:** Planning. The eight asks below were enumerated in `docs/v0.4-findings.md` under "TODO: agent-crash / API-error / hung-session resilience"; this doc is the agreement on shape and sequence.

---

## Why resilience, and why now

Outside feedback on the v0.5 trajectory landed bluntly: *each new round needs to produce output that compounds, not just commits.* `suggest` made bosun newcomer-friendly; the v0.5 polish closed the obvious kickoff gaps. The next compounding step is not another feature surface — it's the safety contract that lets bosun leave the maintainer's machine.

Three things drive the urgency:

1. **The real-repo trial is queued behind v0.6.** Pointing bosun at someone else's codebase (test-only, not shipping) is the moment failure modes go from theoretical to operational. Better to land the resilience layer before that trial than to discover the gaps when a stranger does.
2. **The public launch is queued behind the real-repo trial.** Per the release stance in `CLAUDE.md`, the repo doesn't go public until the tool is well-tested enough that a Hacker News landing experience is good. Resilience is the load-bearing piece of "well-tested enough."
3. **Today's safety is almost entirely git-worktree isolation.** That's a real protection — main is unreachable from a session — but it doesn't help when bosun itself acts on a session that's in a bad state. The v0.6 work fixes the gap between "main is safe" and "bosun won't make things worse when something already went sideways."

What bosun already protects against is captured in `docs/v0.4-findings.md`; this roadmap is the residual.

## The v0.6 asks

Eight items, sequenced by risk-of-corruption (highest first) so the most load-bearing work lands first regardless of which lanes finish.

### 1. Agent-liveness gate on destructive ops

`bosun merge`, `bosun remove`, and `bosun cleanup` must refuse — or require `--ignore-running` — when a session's worktree has a live agent process AND uncommitted dirty files. Today an operator can `bosun merge` a session whose agent is mid-edit; the unstaged changes are silently dropped from the squash. `internal/proc.Running()` already exists from v0.3 work; the v0.6 piece is wiring it into the destructive-op planners and a clear refusal message naming the recovery actions.

**Why first:** highest-blast-radius failure mode. Silent data loss on the operator's most-used command path.

### 2. Pre-merge worktree fsck

Before `git merge --squash <session>`, run `git -C <worktree> fsck --no-progress --no-dangling`. Cheap on the small repos bosun targets (~50ms in practice), and it catches the torn-write case where a disk-full or OOM left a corrupt object that `git status` happily reports as clean. Failure produces a structured error with the fsck output and refuses the merge.

**Why second:** same blast radius as #1 (silent main-branch corruption) but narrower trigger condition.

### 3. Reflog-based merge undo

`bosun merge` records the pre-merge main HEAD into `.bosun/merges.log` (timestamp, session, pre-SHA, post-SHA). A new `bosun merge --undo <merge-id>` resets main to the pre-merge SHA and restores the session's state file. Not strictly necessary — `git reflog` already covers the recovery — but the operator who just lost an hour of work doesn't want to learn reflog under stress. The README points at `--undo` as the first recovery step.

**Why third:** safety net for the cases #1 and #2 don't catch. Cheap to build, expensive when it's missing.

### 4. CRASHED state + `bosun rescue`

A session whose agent process died and whose worktree has uncommitted changes is currently indistinguishable from "WORKING with the agent paused." v0.6 adds a **CRASHED** state derived from `proc.Running() == false && worktreeDirty == true`. `bosun status` surfaces it; `bosun rescue session-N` either (a) snapshots the dirty worktree to `.bosun/rescues/session-N-<timestamp>/` for manual inspection, or (b) pops a launcher window with the half-finished state for the operator to take over. No automated remediation — rescue is operator-driven.

**Why fourth:** the failure mode the real-repo trial will most likely hit. An API 500 mid-edit, a Ghostty window force-close, a sleep-wake cycle — all land here.

### 5. Heartbeat / staleness detection

A new MCP tool `bosun_heartbeat()` the agent calls periodically (or that piggybacks on existing tool calls — `bosun_status` is the obvious carrier). The session record keeps a last-seen timestamp; a session marked RUNNING with no heartbeat for >5 min surfaces as **STALE** in the TUI (yellow) and CLI ("session-X appears stalled — check manually"). No destructive action; just visibility.

**Why fifth:** lower-blast-radius than CRASHED detection (a stalled session usually recovers on its own) but the visibility prevents the operator from wasting cycles debugging "is the agent thinking or is it dead?"

### 6. Hook timeout enforcement

`hooks.Hook` already has a `TimeoutSeconds` field. The v0.6 task is verifying it's actually enforced in `hooks.Run`, defaulting to 30s when zero (zero-means-infinite is the wrong default), and making a hung pre-merge or pre-cleanup hook fail with a clear "hook timed out" error rather than blocking bosun forever. Includes a regression test that wires a `sleep 60` hook and asserts `bosun merge` returns within the timeout.

**Why sixth:** lower-frequency than the agent failure modes, but exactly the kind of thing that surfaces when bosun is wired into someone else's CI-style hooks.

### 7. Init resumability

`bosun init` writes per-step state into `.bosun/init.state` (which session-N was being created, which step completed). A new `bosun init --resume` continues where a prior init left off rather than starting fresh against partial state. Today, init failing partway through — operator force-closes, OS sleeps, disk-full mid-write — leaves orphan branches and half-populated worktrees that the operator has to clean up by hand. We've hit this twice in dogfood.

**Why seventh:** lower-blast-radius (no data loss, just operator friction) but high-frequency enough to be worth the small implementation cost.

### 8. The external-repo safety contract

The README gets a section spelling out exactly what bosun does without operator intervention, what it does only on explicit command, and what it never does:

- **Without explicit command:** creates branches under the `bosun/` prefix; creates worktrees in sibling dirs (`<repo>-bosun-<session>`); writes state to `.bosun/`; binds an MCP socket to `.bosun/mcp.sock`.
- **Only on explicit command:** `merge` (touches main); `remove` / `cleanup` (deletes branches + worktrees); `cleanup --purge` (discards committed work; deliberately loud).
- **Never:** touches main without an explicit merge invocation; writes anywhere outside `<repo>` or `<repo>-bosun-*`; pushes to any remote; modifies git config beyond what's needed for worktrees.

Make the boundaries explicit so a user pointing bosun at their employer's repo can read the contract and decide. This is documentation, not code — but it's load-bearing documentation for the public launch.

**Why eighth:** independent of every other ask; can land alongside any of them. Doesn't move the resilience needle technically, but it's what a new user actually reads to decide whether to trust the tool.

## What's explicitly NOT in v0.6

These came up in the resilience brainstorm and are staying parked:

- **Full transaction-log replay.** Don't store enough state to reconstruct an arbitrary session's history; just enough to recover from a crash. The reflog-undo for merges and the snapshot-on-rescue for sessions are the operational ceiling.
- **Sandbox isolation beyond git worktrees.** Don't try to containerize agent processes. The git-worktree model is the boundary; if an agent runs `rm -rf` on its worktree, the operator's main repo is still fine.
- **Distributed bosun across machines.** Single-machine assumption stays. Multi-machine is a v1.0+ research project, not a v0.6 ask.
- **Continuous health-monitoring daemon.** `bosun doctor` (from v0.5's deferred list) is a one-shot preflight; v0.6 doesn't add a long-running watcher.
- **Net-new feature surface.** No `bosun suggest` v2, no LSP-driven predict, no plugin system. The compounding-not-commits feedback is the explicit principle; v0.6 lands the safety contract and nothing else.

## Sequence

Four-lane round shape, with the hardest-to-merge work front-loaded:

1. **Lane 1 — agent-liveness gate (ask #1).** `cmd/bosun/cmd_merge.go` + `cmd_remove.go` + `cmd_cleanup.go` + planner integration. Tests in `scenarios_test.go`.
2. **Lane 2 — pre-merge fsck + reflog undo (asks #2 + #3).** Pair: both touch `cmd_merge.go` and `internal/git/`. Single lane avoids merge conflict with itself.
3. **Lane 3 — CRASHED state + rescue + heartbeat (asks #4 + #5).** State-machine work in `internal/session/`, MCP tool in `internal/mcp/`, CLI wiring. Largest lane.
4. **Lane 4 — hook timeout + init resume + safety contract (asks #6 + #7 + #8).** Independent surfaces: hooks runner, init planner, README. Parallelizes cleanly.

Each lane is dogfood-shaped per the v0.4-v0.5 pattern. The maintainer keeps the murdge lane open as the round runs.

## After v0.6

1. **Real-repo trial.** Point bosun at an external repo (test-only, not shipping). Capture the punchlist of what surfaces — every "the agent in session-3 hit an API 500 and bosun didn't notice for 20 minutes" entry becomes a v0.6.1 fix.
2. **v0.1 public launch.** Apache 2.0, GitHub visibility flip, Homebrew formula, demo GIF, blog post. Per `CLAUDE.md` release stance, **only** after the trial validates the resilience contract holds in someone else's codebase.

Resist the urge to expand v0.6's scope. The point of the anchor is to stop adding features long enough to make the existing surface trustworthy.

## Open questions

1. **CRASHED detection cadence.** Does bosun poll `proc.Running()` periodically, or only on `bosun status` / `bosun list` invocations? Lazy detection is cheaper; eager catches the case where the operator hasn't run a command in 20 minutes.
2. **Heartbeat collection.** Is `bosun_heartbeat()` a separate tool the agent must explicitly call, or does the existing `bosun_status` / `bosun_claim` traffic implicitly bump the timestamp? Piggyback is simpler for agents already wired up; explicit is clearer for new integrators.
3. **Rescue snapshot retention.** `.bosun/rescues/` accumulates indefinitely without cleanup. Add `bosun rescue --gc` or expire automatically? Operator-driven cleanup is safer.
4. **Reflog-undo and squash specifics.** `git reset --hard <pre-SHA>` is the obvious implementation, but if main has advanced past the recorded SHA (other merges happened since), `--undo` needs to be smarter or refuse. Default to refuse with a clear message; let the operator drive recovery.
