# Bosun v0.6 round 1 — resilience anchor

Four parallel sessions landing the v0.6 resilience layer per
[`docs/v0.6-roadmap.md`](docs/v0.6-roadmap.md). This is the round
that gates the real-repo trial; everything here exists so bosun
doesn't corrupt an operator's codebase when an agent crashes, a
hook hangs, or the API drops a 500 mid-task.

After this round merges, v0.6 has shipped its anchor items
(agent-liveness gate, pre-merge fsck, reflog merge undo, CRASHED
state + rescue, heartbeat, hook timeout, init resumability, safety
contract). The next step is the **external-repo trial** —
test-only, not shipping any code — followed by the public launch.

**Workflow for every session** (in addition to the standard preamble):

1. Implement the work in your lane. Keep diffs scoped.
2. Tests for every new behavior. Race-clean is non-negotiable.
3. Run `make check` before `bosun done`.
4. Findings outside your lane go in the done message, not in code.

**Useful context for every session:**

- The resilience asks in this round are detailed in
  `docs/v0.6-roadmap.md`. Each session below maps to a numbered ask
  in that doc; read the relevant section before starting.
- `internal/proc.Running(worktreePath)` already exists from v0.3
  work and returns `(pid, ok, err)`. Sessions 1, 2, and 3 all
  consume it; nobody re-implements it.
- The session-state enum lives in `internal/session/session.go`:
  `StateWorking`, `StateDone`, `StateStuck`. **Session-3 owns
  adding `StateCrashed`** — the other lanes must not touch this
  file. If your lane needs to reason about whether a session is
  crashed, treat absence of `StateCrashed` as "not crashed" and
  let session-3's merge add it.
- `cmd/bosun/scenarios_test.go` is the standard append-at-EOF
  conflict surface every round hits. All four sessions will
  append; the operator resolves the trivial EOF conflicts at
  merge time. Don't try to be clever about ordering — just
  append at the end with your own clearly-named test functions.

## session-1

**Merge safety — `cmd/bosun/cmd_merge.go` + new `internal/git/fsck.go` + `.bosun/merges.log`.**

This is the v0.6 ask #1, #2, and #3 from the roadmap, all in the
merge command path. Combining them in one lane avoids fighting
session-2 for `cmd_merge.go`.

**Agent-liveness gate (ask #1, merge half).** Before `git merge
--squash` runs for a session, check both:

- `proc.Running(<worktree>)` — is a live agent process rooted in
  this worktree?
- `git -C <worktree> status --porcelain` — does the worktree have
  uncommitted dirty files (modified or untracked)?

If **both** are true, refuse the merge with a clear error:

```
bosun: session-N has a live agent (pid 12345) and uncommitted changes — refusing merge
       to avoid dropping in-flight work. Recovery:
       - finish the agent's edit and `bosun done` from inside the session, OR
       - bosun merge session-N --ignore-running (drops untracked + unstaged work)
```

Add a `--ignore-running` flag to `cmd_merge.go` that bypasses the
gate. Exit code 1 (user error) when refused.

**Pre-merge worktree fsck (ask #2).** Before the squash, run
`git -C <worktree> fsck --no-progress --no-dangling` via a new
helper `internal/git/fsck.go` (function: `FsckWorktree(ctx, dir)
error`). Cheap on small repos (~50ms). If fsck reports any
errors, refuse the merge with the fsck output embedded in the
error. No `--ignore-fsck` flag — fsck failure means torn writes
or corruption, and the operator needs to investigate, not
override.

**Reflog-based merge undo (ask #3).** Before the squash:

- Read main's current HEAD via `git rev-parse HEAD`.
- After a successful merge, append a JSON line to
  `.bosun/merges.log`:
  ```json
  {"ts":"2026-05-16T...","session":"session-N","pre":"<SHA>","post":"<SHA>","squash_msg":"<truncated>"}
  ```
- Add `bosun merge --undo <merge-id>` where merge-id is either the
  session name (newest matching merge wins) or a `<SHA>` prefix.
  Implementation: `git reset --hard <pre-SHA>` on main IF main's
  current HEAD equals the recorded post-SHA. If main has advanced
  since (more merges landed), refuse with a clear "main has moved
  past <session>'s merge; recover via `git reflog`" message. Don't
  try to be smart about cherry-picking.
- Add `bosun merge --list-undo` that prints the recent merge log
  for operator inspection.

**Tests** in `cmd/bosun/cmd_merge_test.go` (or `scenarios_test.go`
append, your call — `cmd_merge_test.go` if it doesn't exist yet
is cleaner):

- Refuses merge when worktree has live agent + dirty files.
- `--ignore-running` bypasses the gate.
- Refuses merge when fsck fails (use a known-bad object fixture
  or a `chmod`-broken `.git/objects/` to trigger).
- Records merges.log entry on successful merge.
- `--undo` resets main to the pre-SHA when main hasn't advanced.
- `--undo` refuses when main has advanced past the recorded
  post-SHA.

- `cmd/bosun/scenarios_test.go` (append at EOF for end-to-end scenarios)

## session-2

**Remove/cleanup safety — `cmd/bosun/cmd_remove.go` + `cmd/bosun/cmd_cleanup.go`.**

The remove+cleanup half of ask #1's agent-liveness gate. Same
pattern as session-1's merge gate, applied to the other two
destructive commands.

**`bosun remove <session>`** must refuse when:

- `proc.Running(<worktree>)` returns a live PID **AND**
- `git -C <worktree> status --porcelain` reports dirty files.

Add `--ignore-running` to bypass. Error message matches session-1's
shape:

```
bosun: session-N has a live agent (pid 12345) and uncommitted changes — refusing remove
       to avoid losing in-flight work. Recovery:
       - let the agent finish, OR
       - bosun remove session-N --ignore-running (discards uncommitted work)
```

**`bosun cleanup`** runs across all sessions; the gate applies
per-session. A session that fails the liveness check is skipped
with a clear "session-N: skipped — live agent with dirty work
(use --ignore-running to override)" line. The rest of the cleanup
proceeds normally. `--ignore-running` is a global flag for cleanup
that applies to all sessions in the batch.

**Important: do NOT change `--purge` semantics.** `--purge` (from
v0.5 r2) handles the DONE-but-unmerged-commits case; the
liveness gate handles the live-agent-with-dirty-files case.
They're orthogonal. A session can need both (`--ignore-running
--purge`) to be removed.

**Tests** in `cmd/bosun/cmd_remove_test.go` and
`cmd/bosun/cmd_cleanup_test.go` (modify both):

- `bosun remove` refuses when live agent + dirty.
- `bosun remove --ignore-running` proceeds.
- `bosun cleanup` skips live+dirty sessions, processes others.
- `bosun cleanup --ignore-running` processes everything.
- `--purge` and `--ignore-running` compose correctly.

- `cmd/bosun/cmd_remove.go` (modify)
- `cmd/bosun/cmd_remove_test.go` (modify)
- `cmd/bosun/cmd_cleanup.go` (modify)

## session-3

**CRASHED state + `bosun rescue` + heartbeat — `internal/session/session.go` + new `cmd/bosun/cmd_rescue.go` + new `internal/mcp/tool_heartbeat.go`.**

Asks #4 and #5 from the roadmap. Session-3 is the largest lane;
it touches the session-state enum, adds a new CLI command, and
adds an MCP tool. All independent of the merge-safety lanes by
file.

**CRASHED state (ask #4, detection half).** Add
`StateCrashed State = "CRASHED"` to the enum in
`internal/session/session.go`. The state is derived during
`session.Derive` when **both**:

- `proc.Running(<worktree>) == false` (no live agent) **AND**
- `git -C <worktree> status --porcelain` reports dirty files.

A WORKING session that meets both conditions becomes CRASHED in
the derived output. DONE sessions are never CRASHED. `bosun status`
and `bosun list` already iterate Derive's output; they'll pick up
the new state automatically. Update the status renderer to color
CRASHED red.

**`bosun rescue <session>` (ask #4, recovery half).** New
subcommand in `cmd/bosun/cmd_rescue.go`. Two modes:

- **Default mode (snapshot):** copies the dirty worktree contents
  (the modified + untracked files only, NOT the whole worktree) to
  `.bosun/rescues/session-N-<timestamp>/`. The original worktree
  is left intact. Prints the snapshot path so the operator can
  inspect.
- **`--launch` mode:** pops a Ghostty/iTerm2 window in the
  worktree so the operator can take over the half-finished state
  manually. Reuses the existing launcher logic from `cmd_launch.go`.

Refuses (exit 1) if the session isn't in CRASHED state — rescue
is for the specific recovery case, not a general escape hatch.

Wire `bosun rescue` into `cmd/bosun/root.go` (one-line
`newRescueCmd()` registration).

**Heartbeat (ask #5).** New MCP tool `bosun_heartbeat` in
`internal/mcp/tool_heartbeat.go`. Takes the session label as
argument; writes the current timestamp into
`.bosun/state/session-N.heartbeat` (with the same cross-process
flock pattern session-2 of v0.3.1 established for state files).

Add a STALE derived state in `session.Derive`: a WORKING (not
CRASHED) session whose heartbeat is older than 5 minutes
surfaces as STALE. STALE is NOT a separate enum constant — it's
a derived flag on the Session struct (e.g. `Stale bool`) so the
existing state enum stays compact and CRASHED remains the only
new state value. The status renderer shows STALE as yellow text
alongside the state column.

Don't piggyback heartbeats on existing tool calls — that
complicates the contract. A separate tool is clearer and lets
the agent choose its own cadence.

**Tests** in `internal/session/session_test.go` (CRASHED + stale
derivation), `cmd/bosun/cmd_rescue_test.go` (new, rescue
behaviors), `internal/mcp/tool_heartbeat_test.go` (new):

- CRASHED detection when proc not running + worktree dirty.
- CRASHED NOT triggered when proc running OR worktree clean.
- Stale flag set when heartbeat older than 5 min.
- Rescue refuses on non-CRASHED session.
- Rescue snapshot creates `.bosun/rescues/...` with modified files.
- Heartbeat tool updates the file with current timestamp.
- Concurrent heartbeat writes don't tear (flock).

## session-4

**Hook timeout + init resumability + safety contract — `internal/hooks/hooks.go` + `cmd/bosun/cmd_init.go` + new `internal/init/state.go` + `README.md`.**

Asks #6, #7, and #8 from the roadmap. Three independent surfaces
that fit naturally into one lane because they share no files with
any other session.

**Hook timeout enforcement (ask #6).** `hooks.Hook` already has a
`TimeoutSeconds` field. Audit `hooks.Run` (the central runner) and
verify the timeout is actually applied via `context.WithTimeout`
around the `exec.CommandContext` invocation. If it isn't, add it.
**Default to 30 seconds when `TimeoutSeconds == 0`** — never let
zero mean infinite. A timed-out hook returns a structured error
`hooks.ErrTimeout` that the caller renders as "hook <name> timed
out after Xs".

Add a regression test in `internal/hooks/hooks_test.go` that
configures a `sleep 60` hook and asserts the runner returns
within ~31 seconds with `ErrTimeout`. The test can use a shorter
timeout (e.g. 2s) by setting `TimeoutSeconds` explicitly so it
doesn't actually wait 30 seconds.

**Init resumability (ask #7).** `bosun init` writes per-step state
into a new `.bosun/init.state` JSON file as it progresses:

```json
{
  "version": "v0.6",
  "started_at": "2026-05-16T...",
  "plan_path": "v0.6-round1-plan.md",
  "total_sessions": 4,
  "completed_sessions": ["session-1", "session-2"],
  "current_session": "session-3",
  "current_step": "git_worktree_add"
}
```

Steps: `branch_create`, `git_worktree_add`, `state_file_write`,
`hook_post_init`. On success of all N sessions, the file is
deleted.

Add `bosun init --resume`:

- If `.bosun/init.state` exists, parse it and continue where it
  left off.
- Sessions already in `completed_sessions` are skipped (assume
  they exist on disk; verify with a cheap `git worktree list`
  check and refuse with a clear error if they don't).
- The current session restarts from its `current_step` if the
  step is idempotent, otherwise from the start.
- Plain `bosun init` without `--resume` refuses to run if
  `.bosun/init.state` exists ("previous init didn't finish; run
  `bosun init --resume` or `rm .bosun/init.state` to start
  fresh"). This is the safety gate against the "operator
  doesn't realize init half-failed" case we hit twice in
  dogfood.

Move the init step machinery into a new `internal/init/state.go`
package so the state file format is testable in isolation:

- `type InitState struct { ... }`
- `func Load(repoRoot string) (*InitState, error)`
- `func (s *InitState) Save(repoRoot string) error`
- `func (s *InitState) MarkComplete(label string) error`
- `func (s *InitState) Clear(repoRoot string) error`

`cmd_init.go` consumes the new package.

**Safety contract (ask #8).** Add a new top-level section to
`README.md` titled **"Safety contract — what bosun does to your
repo"**. Three subsections:

- **Without explicit command, bosun:** creates branches under the
  `bosun/` prefix; creates worktrees in `<repo>-bosun-<session>`
  sibling directories; writes state under `.bosun/`; binds an
  MCP socket at `.bosun/mcp.sock`.
- **Only on explicit command, bosun:** `merge` (touches main);
  `remove` / `cleanup` (deletes branches + worktrees); `cleanup
  --purge` (discards committed work, deliberately loud); `merge
  --undo` (resets main to a prior SHA, only if main hasn't
  advanced).
- **Never, regardless of command, bosun:** touches main without
  an explicit merge invocation; writes outside `<repo>` or
  `<repo>-bosun-*`; pushes to any remote; modifies global git
  config; modifies repo-level git config beyond what's needed
  for worktrees.

Place the section after the "Quick start" but before the
"Architecture" section in the README. Keep it under 50 lines —
this is a contract, not a manual.

**Tests** for the hook timeout (above) and for the init state
package (`internal/init/state_test.go`):

- Load/Save round-trip.
- MarkComplete persists across reload.
- Clear removes the file.
- Concurrent writes don't tear (flock — same pattern as
  state.go uses for session state files).

Init resumability needs a scenarios test in
`scenarios_test.go`:

- Run `bosun init 3` against a fake `git worktree add` that
  fails on the 2nd session.
- Verify `.bosun/init.state` exists with session-1 in
  `completed_sessions`, session-2 in `current_session`.
- Re-run `bosun init --resume`; verify session-2 + session-3
  are created and `.bosun/init.state` is removed.

## Sequencing notes (for the operator)

- Kickoff: `bosun init 4 --brief v0.6-round1-plan.md` (no
  `--launch`; manual `bosun launch session-N` per session to
  avoid the system-load spike from 4 simultaneous Ghostty windows).
- Four disjoint lanes by file:
  - session-1 owns `cmd_merge.go` + `internal/git/fsck.go`
  - session-2 owns `cmd_remove.go` + `cmd_cleanup.go`
  - session-3 owns `internal/session/session.go` + `cmd_rescue.go` + `internal/mcp/tool_heartbeat.go`
  - session-4 owns `internal/hooks/hooks.go` + `cmd_init.go` + `internal/init/` + `README.md`
- Standard merge surfaces: `cmd/bosun/scenarios_test.go` (all
  four append at EOF) and `cmd/bosun/root.go` (session-3 adds
  `newRescueCmd()` registration).
- No `depends:` ordering required — all four lanes are
  independent. The merge can land them in any order.
- **Lessons from prior rounds to internalize:**
  - When a shared file (`internal/session/session.go`) needs
    extension, exactly ONE lane owns the file (session-3 here).
    The other lanes consume the existing surface unchanged and
    must not edit it. The v0.5 round-3 disaster was two lanes
    racing on `internal/predict/types.go`; this round is
    explicit about ownership.
  - Don't run `bosun cleanup --force` while sessions are
    DONE-but-unmerged. Use `bosun merge` first, or
    `bosun cleanup --purge` for the deliberate discard case.
  - Re-merging hand-resolved squashes can re-conflict; clear
    `.bosun/state/session-N.done` after a successful manual
    commit, OR use `bosun cleanup --purge` once the resolved
    commit is on main.
  - `proc.Running()` is read-only territory for sessions 1, 2,
    and 3. Nobody edits `internal/proc/detect.go` — extend
    behavior in your own package if you need it.
