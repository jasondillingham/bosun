# Bosun external-repo trial protocol

**Goal:** validate the v0.6 resilience contract in a codebase bosun
has never seen. This is the gate between v0.6 and the public v0.1
launch per [`CLAUDE.md`](../CLAUDE.md) release stance.

**Scope:** test-only. No code from the trial repo ships anywhere; no
merges to the trial repo's upstream/main. Bosun creates its own
`bosun/<session>` branches and worktrees; everything stays local.

**What "success" looks like:** bosun runs a 2-4 lane round on the
trial repo, the safety gates fire when they should, the operator
finishes the trial without losing any of the trial repo's pre-existing
work, and the surfaces that need polish are written down.

---

## Pre-trial setup

### 1. Binary on PATH

The installed bosun lives at `~/go/bin/bosun`. Confirm version
matches v0.6:

```sh
~/go/bin/bosun --help | head -3
~/go/bin/bosun config show 2>&1 | head -3
```

If `~/go/bin` isn't on the interactive PATH, either:

- Add to `.zshrc`: `export PATH="$HOME/go/bin:$PATH"`
- Or symlink: `sudo ln -s ~/go/bin/bosun /usr/local/bin/bosun`

The trial uses the absolute path `~/go/bin/bosun` throughout this doc
so it works regardless.

### 2. Pick a target repo

Criteria:

- **You own it.** No one else's main branch in scope.
- **Real shape.** Not a toy. Has src files, tests, package boundaries.
- **Low-stakes.** OK if a trial round burns 10 min you have to back out.
- **Not bosun itself.** The whole point is "external."
- **Reasonably small.** A few hundred to a few thousand src files; not
  a node_modules-bloated monolith (those mask filesystem signals).

Reasonable candidates from `~/Documents/Homelab/`:

- `homelab-status-mcp` — small Go MCP server, real code, parallelizable
- `architect-mcp` — same shape, slightly larger
- `ollama-mcp` — same shape
- `personal-site` — static site generator, smaller surface
- `jasondillingham-dev` — only if it isn't deployed-on-push

**Avoid:** `jobhunt-dashboard-next` (Next.js, ~12k files in
node_modules — bosun will inherit that filesystem pressure and the
trial signal gets muddy). Any repo currently mid-deploy.

### 3. Pre-flight on the operator side

Before running `bosun init` on the trial repo:

- `uptime` — 1-min load avg < 5
- `df -h ~` — > 20G free
- Close any in-flight Claude Code sessions for the trial repo
- Commit or stash anything pending in the trial repo
- `cd <trial-repo> && git status` — must be clean

### 4. Pick a trial task

The task itself is secondary — what matters is exercising bosun. Pick
something:

- **Genuinely parallelizable** (2-4 disjoint lanes; not "refactor
  everything")
- **Bounded** (each lane < 30 min of agent work)
- **Reversible** (worst case: `bosun cleanup --purge` and the trial
  repo is untouched)
- **Real but disposable** (you don't actually need the output — the
  point is to watch bosun, not ship a feature)

Examples that fit:

- "Add tests for 3 named packages, one per session"
- "Document the public API of N modules in their package doc comments"
- "Convert N files from one logging library to another"
- "Add a CLI flag `--foo` to N subcommands"

Examples that don't:

- "Refactor the architecture" (too vague, lanes will overlap)
- Anything touching shared types/config files
- Anything with external service calls (slow, flaky, not bosun signal)

## During the trial

### Watch list — green flags

These are the v0.6 features doing their jobs. Note when they fire:

- `bosun status` shows a CRASHED session after you Ctrl-C'd a worktree
  with dirty edits
- `bosun merge` refuses with "live agent + uncommitted changes" when
  you try to merge a session whose agent is mid-edit
- `bosun rescue session-N` produces a snapshot under `.bosun/rescues/`
- A hook with `TimeoutSeconds: 0` doesn't hang forever (defaults 30s)
- `bosun merge --undo session-N` resets main correctly when main
  hasn't advanced
- `.bosun/init.state` exists during a long init; `bosun init --resume`
  picks up cleanly after a forced kill

### Watch list — red flags (HARD STOP)

If any of these happen, **stop immediately** and capture state for the
findings doc. Don't try to recover by running more bosun commands;
preserve the broken state.

- **Bosun touches anything outside `<repo>` or `<repo>-bosun-*`.**
  This is the safety-contract violation. Filesystem boundary check via
  `find ~ -newer <a-timestamp-marker> -type f` if needed.
- **Bosun pushes to a remote.** Should never happen; if `git log
  origin/main..HEAD` shows the trial repo's `origin` has new commits
  bosun put there, that's a contract break.
- **Main branch advances without an explicit `bosun merge` you ran.**
  Watch `git rev-parse main` before and after each bosun operation.
- **A merge silently drops uncommitted work** despite the liveness
  gate (i.e. you had a dirty worktree + live agent and bosun merged
  anyway).
- **`bosun cleanup` deletes a session whose commits aren't on main and
  you didn't pass `--purge`.** The whole point of the v0.5 r2 gate.
- **Init failure leaves orphan branches you have to manually
  delete.** The init.state machinery should make `--resume` clean.
- **Any agent process attempts to write outside its own worktree.**
  Less bosun's fault, more agent-discipline, but log it.

### Per-operation discipline

For every bosun command you run during the trial:

1. **Before:** `git status` in the trial repo + capture `git rev-parse
   main` to a scratch file.
2. **Run the command.**
3. **After:** `git status` in the trial repo + diff `git rev-parse
   main`. Confirm nothing changed outside the expected blast radius
   for the command you ran.

This is tedious. Do it anyway for the first 3-4 commands; once you've
seen the contract hold, you can relax.

## Trial flow (suggested)

### Phase 1 — solo session shakedown (10 min)

Goal: verify bosun's basic loop works in the trial repo before
spawning multiple agents.

```sh
cd <trial-repo>
~/go/bin/bosun init 1
~/go/bin/bosun status
~/go/bin/bosun launch session-1 --initial-prompt "Read BOSUN_BRIEF.md..."
# Let it do one tiny task end-to-end.
# Watch bosun status. Run bosun_check from inside the session.
# When agent is done, bosun merge session-1.
# Then bosun cleanup.
```

If Phase 1 hits any red flag, stop. v0.7 needs that bug fixed first.

### Phase 2 — multi-session round (20-30 min)

```sh
# Write a plan markdown with 2-3 disjoint lanes (see "trial task"
# section above). Save to <trial-repo>/trial-plan.md.
~/go/bin/bosun init 3 --brief trial-plan.md
~/go/bin/bosun launch session-1 --initial-prompt "Read BOSUN_BRIEF.md..."
~/go/bin/bosun launch session-2 --initial-prompt "Read BOSUN_BRIEF.md..."
~/go/bin/bosun launch session-3 --initial-prompt "Read BOSUN_BRIEF.md..."
```

While they run, exercise the resilience gates deliberately:

- Pick one session and force-close its Ghostty window. Run `bosun
  status` — expect CRASHED. Run `bosun rescue session-N` — expect a
  snapshot dir.
- Pick another session and try `bosun merge session-N` while the
  agent is mid-edit (dirty worktree). Expect refusal with the live-
  agent message.
- Let the third session finish normally. Run `bosun merge session-N`,
  then `bosun merge --undo session-N`, then verify main is back to the
  pre-merge SHA.

### Phase 3 — teardown

```sh
~/go/bin/bosun cleanup           # safe sessions only
~/go/bin/bosun cleanup --purge   # discard the rest
ls .bosun/rescues/               # confirm rescue snapshots survived
# Optionally inspect / delete .bosun/ after capturing findings.
```

## Post-trial — findings capture

Write a new `docs/v0.6-trial-findings.md` with:

- **What worked.** Each green-flag observation, with a one-line "this
  is the v0.6 ask for X firing correctly."
- **What surprised you.** Anything that worked but felt awkward.
  Naming, error message phrasing, output verbosity, missing flags.
- **What broke.** Each red flag in detail. Include the exact command
  sequence, the trial repo state, and what bosun produced.
- **What's missing.** Surfaces the trial revealed bosun doesn't have
  yet. These become the v0.6.1 punchlist or v0.7 candidates.

Frame the findings as the gate to the public launch. Per
`CLAUDE.md`'s release stance: bosun goes public only when "a new user
landing on the GitHub page from a Hacker News thread has a good
experience." The trial is the empirical evidence either direction.

## Out of scope for the trial

These aren't the trial's job. Don't bend the trial to cover them:

- **Performance benchmarking.** Single-machine wall-clock vs. serial.
  Separate exercise.
- **Cross-OS validation.** macOS trial is the v0.1 target; Linux
  trial is v0.2's job.
- **Multi-user collab.** Single operator throughout.
- **Long-running agents.** The trial is hours, not days. Reliability
  over weeks is a v1.0 question.
- **Public-launch artifacts.** No README rewrites, demo GIFs, or HN
  posts during the trial. Per release stance, those wait for explicit
  go-ahead.

---

## If the trial uncovers a contract break

`CLAUDE.md`'s release stance is the load-bearing gate here. A
contract break — bosun touching main without explicit command,
silently dropping work, etc. — means:

1. **Stop the trial.** Preserve the broken state.
2. **Write the bug to `docs/v0.6-findings.md`** with full reproducer.
3. **Don't ship v0.1.** v0.6.1 (or v0.6.2, however many rounds it
   takes) addresses every contract break before the trial resumes.
4. **Re-run the trial** from Phase 1 against the fixed bosun.

This is exactly the failure mode the trial exists to catch. A clean
trial is the gate; a dirty trial is the input that produces the next
round of work.
