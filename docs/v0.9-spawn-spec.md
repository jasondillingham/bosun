# v0.9 spawn spec — `bosun_spawn` MCP tool

The design document for the v0.9 agent-driven coordination work.
Lane 1 of `docs/v0.9-roadmap.md`. Lanes 2-4 implement against this
doc.

The point of writing the spec before any code: the design choices
about auth, hierarchy, and termination ripple through the MCP tool,
the status renderer, and the cleanup machinery. Getting them
wrong in code costs three lanes; getting them wrong here costs
re-reading the doc.

---

## What the tool does

A Claude session running inside a bosun worktree can call
`bosun_spawn` over the MCP socket to create N sub-sessions —
worktrees + branches + briefs, exactly the way `bosun init` does —
that the parent agent has decided to fan out as parallel sub-tasks.

The operator authorizes the *capability* once via
`.bosun/config.json`; the *trigger* moves to the agent.

---

## Authorization model

Three tiers; v0.9 ships tier 1 only.

### Tier 0 — operator-only (today, unchanged)

`bosun init` from the shell creates top-level sessions. Same as
v0.1-v0.8.

### Tier 1 — agent-spawnable with repo-level opt-in (v0.9)

The operator flips a config gate in `.bosun/config.json`:

```json
{
  "agent_spawn": {
    "enabled": false,
    "max_concurrent_sub_sessions": 3,
    "max_depth": 1,
    "allowed_for_sessions": null
  }
}
```

Defaults (when the config key is absent entirely):

- `enabled: false` — agents cannot spawn until explicitly enabled.
- `max_concurrent_sub_sessions: 3` — per-parent cap on live sub-
  sessions. A parent that's already at the cap gets a tool error.
- `max_depth: 1` — sub-sessions of a parent cannot themselves spawn
  by default. Depth 2 means grandchildren are allowed; defaults to
  the lower number for safety. **Hard ceiling: 4.** Configured
  values above 4 are silently clamped to 4 to bound fork-bomb risk.
- `allowed_for_sessions: null` — when null, any session may spawn.
  When set to a list of label patterns, only those sessions may
  spawn. Useful for "this one workflow is allowed to fan out, the
  others stay single-shot."

### Tier 2 — full autonomy (deferred to v1.x or later)

Agents calling destructive operations (`bosun_merge`, `bosun_remove`)
without operator-side gating. Out of scope for v0.9.

## Branch model: hierarchical

Sub-sessions branch **from the parent's branch**, not from main.

Rationale: the motivating use case is "agent realizes its work would
parallelize cleanly across N sub-tasks AND wants to integrate the
results." That requires the sub-sessions to see the parent's
already-completed work as their base, otherwise the agent is
duplicating context-setting in each sub-brief.

Branch graph for a spawn of `session-1` → `session-1.auth` +
`session-1.http`:

```
main
 └─ bosun/session-1
     ├─ bosun/session-1.auth   (branched from session-1's HEAD at spawn time)
     └─ bosun/session-1.http   (branched from session-1's HEAD at spawn time)
```

Merge flow:

1. Sub-sessions merge to parent's branch (each via the normal
   `bosun merge` against parent's branch as base).
2. Parent's branch merges to main (normal `bosun merge`).

Constraint enforced by bosun: **`bosun merge session-1` refuses
when session-1 has any unmerged sub-sessions.** Error message
names the subs and points at `bosun merge --tree session-1` (which
merges subs first then the parent in dependency order).

## Label scheme: dot-separated suffixes

Sub-session labels are `<parent>.<suffix>`.

Examples:

- Parent `session-1` spawns subs with suffixes `auth` and `http` →
  `session-1.auth` and `session-1.http`.
- Parent `auth` (named session) spawns sub `handler` → `auth.handler`.
- Grandparent `session-1` spawned `session-1.auth` which spawns
  `parser` → `session-1.auth.parser`. Allowed only when
  `max_depth >= 2`.

The label validator (`internal/session.ValidateLabel`) gets
extended to permit a single `.` between otherwise-valid label
segments. Concretely:

```
existing: ^[a-z][a-z0-9]*(-[a-z0-9]+)*$
v0.9:     ^[a-z][a-z0-9]*(-[a-z0-9]+)*(\.[a-z][a-z0-9]*(-[a-z0-9]+)*)*$
```

A label may contain dots; no leading/trailing dot; no consecutive
dots; each dot-separated segment matches the existing label
charset.

Worktree paths follow the same convention bosun already uses:
`<repo>-bosun-<label>`. So `session-1.auth` gets `<repo>-bosun-session-1.auth`.
Branch names: `bosun/session-1.auth`. Dot is a legal git ref char
(it just can't appear at the start or end of a segment).

## Tool surface

Single MCP tool. Signature:

```go
// MCP tool: bosun_spawn
// Called by an agent running inside a worktree to fan out work
// into sub-sessions.
type SpawnArgs struct {
    // Parent is the calling session's own label. The MCP tool
    // verifies the calling connection's worktree matches this label
    // (no spoofing).
    Parent string `json:"parent"`

    // Brief is the inline markdown brief — same shape `bosun init
    // --brief` consumes. Headings like `## auth` describe the
    // sub-session suffixes; the tool auto-prepends `<parent>.` to
    // each before creating worktrees.
    Brief string `json:"brief"`

    // Launch controls whether the spawn fires `bosun launch` for
    // each new sub-session. Defaults to true; set false to create
    // the worktrees + briefs without launching agents (useful for
    // operator review before agents start).
    Launch bool `json:"launch"`
}

type SpawnResult struct {
    // Created is the list of sub-session labels that were
    // successfully created (worktree + branch + brief).
    Created []string `json:"created"`

    // Failed lists per-label failure reasons for spawns that didn't
    // complete. Possible reasons: quota exceeded, depth exceeded,
    // duplicate label, brief parse error, git failure.
    Failed []SpawnFailure `json:"failed"`
}

type SpawnFailure struct {
    Label  string `json:"label"`
    Reason string `json:"reason"`
}
```

Call semantics:

- Tool is callable only when `agent_spawn.enabled == true` in
  `.bosun/config.json`. Otherwise it returns a clear "agent_spawn
  is not enabled for this repo" error and emits no events.
- Tool verifies the calling agent's worktree corresponds to the
  claimed `Parent` label. This uses the same `proc.Running` +
  worktree-CWD logic that `bosun status` uses for RUNNING column
  detection. If the calling process isn't visible inside the
  parent's worktree, refuse with "spawn requires the caller to be
  the named parent session."
- Quota check: count current sub-sessions in `.bosun/spawn-tree.json`
  with this parent. If at or above `max_concurrent_sub_sessions`,
  refuse with "spawn quota exceeded."
- Depth check: count dot-separated segments in `parent`. If the
  resulting sub-depth would exceed `max_depth`, refuse with "spawn
  depth exceeded."
- Brief parsing: `internal/brief.ParseString(brief)` (new variant
  that takes a string instead of a path) returns the heading list.
  Each heading becomes a suffix; the full sub-label is
  `<parent>.<suffix>`.
- Create each sub-session by delegating to a helper extracted from
  `cmd/bosun/cmd_init.go` (the per-session create pipeline:
  CreateBranch, AddWorktree, WriteBrief, optionally Launch).
- Spawn-tree update: record each successful sub-session in
  `.bosun/spawn-tree.json` (atomic write through the existing
  lockfile package).
- Event emission: write a `session_spawned` event per sub-session
  to `.bosun/events.jsonl` (new event type for the MCP event log).
- Return `SpawnResult` with the created labels and any failures.

## Spawn-tree tracking

New file: `.bosun/spawn-tree.json`. Schema:

```json
{
  "version": "v1",
  "sessions": {
    "session-1": {
      "depth": 0,
      "parent": null,
      "children": ["session-1.auth", "session-1.http"],
      "spawned_at": "2026-05-17T18:30:00Z"
    },
    "session-1.auth": {
      "depth": 1,
      "parent": "session-1",
      "children": [],
      "spawned_at": "2026-05-17T19:05:12Z"
    },
    "session-1.http": {
      "depth": 1,
      "parent": "session-1",
      "children": [],
      "spawned_at": "2026-05-17T19:05:12Z"
    }
  }
}
```

A top-level session has `depth: 0` and `parent: null`. The
`children` list is populated when a session spawns sub-sessions. A
sub-session record is created the moment the spawn succeeds.

Operations:

- Read: any time bosun renders status, plans a cleanup, or refuses
  a merge that has unmerged subs.
- Write: only `bosun_spawn` (creates records) and
  `bosun cleanup`/`bosun remove` (removes records when a session
  is reaped).
- Concurrency: writes go through `internal/lockfile.WithLock` on
  `.bosun/spawn-tree.lock` — same pattern claims/state use.

## Operator visibility

`bosun status` renders tree-shape when any session has children OR
a parent:

```
$ bosun status
4 sessions — 1 DONE, 2 WORKING, 1 CRASHED · 5 commits ahead total
SESSION              BRANCH                       STATE    AHEAD  DIRTY  ...
session-1            bosun/session-1              WORKING  0      2      ...
  ├─ session-1.auth  bosun/session-1.auth         DONE     1      0
  └─ session-1.http  bosun/session-1.http         WORKING  0      1
session-2            bosun/session-2              DONE     1      0
```

Top-level sessions are flush-left; children are indented under
their parent with a tree-prefix.

`bosun status --no-tree` falls back to the flat v0.8 rendering for
scripts that parse the output.

`bosun show <session>` adds two lines when a session is part of a
tree:

```
$ bosun show session-1.auth
session-1.auth (DONE)
  parent: session-1 (WORKING)
  ...
```

```
$ bosun show session-1
session-1 (WORKING)
  spawned children:
    session-1.auth (DONE, 1 commit ahead)
    session-1.http (WORKING, 0 commits)
  ...
```

`bosun list` keeps its flat one-name-per-line output (so scripts
that parse it don't regress) but adds `--tree` for the indented
variant.

Web dashboard: `/api/status` returns the spawn-tree info as a
nested JSON tree (parent.children = []SubSession).

Events log gets a new event type:

```json
{
  "ts": "2026-05-17T19:05:12Z",
  "type": "session_spawned",
  "parent": "session-1",
  "child": "session-1.auth",
  "via": "bosun_spawn"
}
```

TUI receives these events live and refreshes the tree view.

## Recovery semantics

### Parent crashes (agent process exits while sub-sessions are live)

- Sub-sessions continue running. Their agent processes are
  independent.
- `bosun status` shows the parent as CRASHED (per v0.6 semantics)
  but the subs as WORKING / DONE / whatever.
- Operator options:
  - `bosun rescue session-1` (the parent) — snapshots parent's
    dirty files, leaves children alone.
  - `bosun cleanup --tree session-1` — reaps the whole subtree.
  - `bosun adopt session-1.auth` — promotes the named sub to
    top-level (clears its `parent` reference in spawn-tree.json).
    Useful when the operator decides "this sub is the only thing
    worth continuing."

### Sub crashes (agent process exits, sub-session goes CRASHED)

- Parent agent has to poll. `bosun_status` MCP tool already
  returns each session's state; the parent reads its own children
  via `bosun_status` (or a future `bosun_subsessions` if polling
  every status is too noisy).
- Parent decides how to handle: respawn (call `bosun_spawn` again
  with the same suffix — quota allowing), give up (operator-
  driven cleanup), or continue without the sub.

### Operator stops the whole run

- `bosun cleanup --tree session-1` — single command, cascades
  through the subtree. Each sub goes through the standard
  liveness/dirty gate before removal.
- For absolute escape hatch, `bosun cleanup --tree session-1 --force --ignore-running`
  cascades + bypasses the gates. Same shape v0.6 added for the
  flat case.

## Termination rules

- **`bosun merge <parent>` refuses** when the parent has unmerged
  sub-sessions. Error: "session-1 has 2 unmerged sub-session(s);
  merge them first via `bosun merge --tree session-1` or
  individually."
- **`bosun merge --tree <parent>`** merges subs in dependency
  order (subs first; parent last). If any sub fails to merge,
  surfaces the failure and refuses to proceed to the parent.
- **`bosun cleanup <parent>` refuses** when the parent has live
  sub-sessions. Error: "session-1 has 2 live sub-session(s); reap
  them first via `bosun cleanup --tree session-1` or
  individually."
- **`bosun remove <sub>`** works normally — removes one sub
  without touching the parent.
- **`bosun remove <parent>`** refuses when subs exist. Same
  message shape as cleanup's refusal.

## Safety contract additions

`README.md`'s safety-contract section gains one bullet under
"Only on explicit command, bosun":

> When `agent_spawn.enabled` is true in `.bosun/config.json`,
> agents running inside session worktrees can call the
> `bosun_spawn` MCP tool to create sub-sessions. The operator
> authorizes the capability once via that config; bosun bounds
> sub-session count and tree depth per the config's quotas. The
> operator can revoke at any time by setting `enabled: false` and
> reaping outstanding subs via `bosun cleanup --tree`.

Nothing else in the contract changes. Main is still untouched
without an explicit `bosun merge`. Branches still live under
`bosun/`. Worktrees still live in `<repo>-bosun-*` siblings.

## Test surface

Unit tests (per-package, fast):

- `internal/spawntree/` — new package owning the spawn-tree JSON
  schema + read/write/concurrent-write tests.
- `internal/session.ValidateLabel` — new cases for dotted labels:
  `session-1.auth` (valid), `.session-1` (invalid),
  `session-1..auth` (invalid), `session-1.` (invalid).
- `internal/mcp/tool_spawn_test.go` — tool semantics: quota
  enforcement, depth enforcement, brief parsing, label collision
  detection, parent-spoofing rejection.
- `internal/config` — `agent_spawn` section parsing, default
  values, clamping of `max_depth` above 4.

Scenarios (slower, end-to-end):

- A real spawn via the MCP socket end-to-end: parent agent
  authenticates, calls `bosun_spawn` with a 2-sub brief, both subs
  land on disk, status renders the tree, `bosun merge --tree
  session-1` runs all three merges in order.
- Quota exceeded: parent already has `max_concurrent_sub_sessions`
  subs alive; another spawn fails cleanly.
- Depth exceeded: a session with `depth == max_depth` calling
  spawn fails with a clear error.
- Adopt: sub-session adopted to top-level after parent reap.

External trial #3 (after lanes 2-4 land):

- Same `homelab-status-mcp` baseline as trials 1+2.
- Parent agent spawns 2 subs. Parent runs to DONE, subs run to DONE.
- Operator merges via `bosun merge --tree`.
- Safety verification: main HEAD = pre-trial baseline + 3 squash
  commits in dependency order.

## Open issues — fold into lane 2 work or kick to v0.10

- **Agent identity verification.** The tool verifies the caller is
  inside the parent's worktree via `proc.Running`-style CWD
  matching. That's pragmatic but not cryptographic. If a malicious
  process inside a worktree could call bosun_spawn impersonating
  the parent, the worst it could do is create sub-sessions under
  the parent's name (limited by quotas). Acceptable risk for v0.9.
  v1.x could ship a per-session MCP socket token.
- **Cost surfacing.** A 4-sub spawn at v0.6+ rates is $X. v0.9
  doesn't track or display that. Operator monitors via the API
  provider's dashboard. v1.x could add a `bosun status --cost`
  estimate.
- **`bosun init` syntax that pre-creates a tree.** Today `bosun
  init 3 --brief x.md` creates 3 top-level sessions. A future
  shape like `bosun init --tree parent.md` could create a parent
  plus its planned-children all at once, without an agent spawn.
  Useful for operators who know the tree shape up front. v0.10+.
