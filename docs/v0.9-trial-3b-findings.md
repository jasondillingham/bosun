# v0.9 trial #3b findings ‚Äî homelab-status-mcp (2026-05-17)

## Outcome

**Negative evidence on `bosun_spawn`.** The agent ran the full
brief solo and produced excellent work ‚Äî three `Health()`
implementations + an aggregating MCP tool, all in one commit ‚Äî but
never called `bosun_spawn`, even with a brief explicitly engineered
to make spawning the obvious play. The trial's primary goal (exercise
the v0.9 spawn flow under realistic conditions) was not met. Its
secondary goal (validate the safety contract, the launcher, the
hook, and the broader v0.9 surface area) was met.

This is high-signal product feedback: a brief that says "use this
tool" is not enough to overcome an agent's preference for solo work
when the work feels tractable to it.

## Why the agent didn't spawn

spawn-tree.json at end of trial:

```json
{
  "version": "v1",
  "sessions": {
    "session-1": {
      "depth": 0,
      "spawned_at": "2026-05-18T02:49:58Z"
    }
  }
}
```

Zero child sessions. The agent never called `bosun_spawn`.

The brief was direct about it:

> You have access to the `bosun_spawn` MCP tool. **Use it.** The three
> `Health()` implementations are disjoint package-shaped work ‚Äî this
> is exactly the shape spawn was built for.

The brief even gave a step-by-step recipe (call spawn three times,
each with a one-paragraph sub-brief, then `merge --tree`).

Without access to the agent's chain-of-thought, we can only
hypothesize. Plausible reasons:

1. **The work felt tractable solo.** Once the agent had loaded the
   three `client.go` files into its context, doing the three
   `Health()` methods in series was 30 minutes of edits ‚Äî about the
   same wall time the brief estimated for the round end-to-end. The
   cost of spawning (writing three sub-briefs, waiting on merges)
   probably looked higher than the parallelism benefit.
2. **The brief's "warning" framed spawn as opt-in.** The same
   paragraph that said "use it" also said "the point of this trial
   is to exercise the spawn path, but only if the work genuinely
   warrants it. False-spawning is anti-signal." The agent may have
   read that as permission to decide for itself, and decided the
   work didn't warrant fan-out.
3. **MCP-side issue.** Possible the `bosun_spawn` tool wasn't
   visible to the agent at all (BOSUN_MCP_SOCK was set but the MCP
   client might not list the tool, the tool description might be
   too thin to motivate use, etc.). The launcher reported "Started
   MCP server (pid 94359) on /Volumes/SneakerNet/...sock" so the
   socket existed; whether the tool was discoverable from inside
   the agent is unconfirmed without a trace.

## What's not exercised (carried forward to trial #3c)

The spawn-tree machinery downstream of `bosun_spawn` remains
unvalidated end-to-end in an external repo:

- Hierarchical labels in `bosun status` (`session-1.cloudflare`
  etc.) under load
- TUI / web dashboard tree rendering with real sub-sessions
- `bosun cleanup` refusal when parent has live subs
- `bosun cleanup --tree` cascade
- `bosun merge --tree session-1` post-order cascade ‚Äî the parent
  branch never had subs to merge in
- Quota refusal when a 4th `bosun_spawn` would exceed
  `max_concurrent_sub_sessions=3`
- Depth refusal when a sub tries to spawn its own sub (max_depth=1)

A trial #3c needs a brief that's even harder to do solo. Options:

- **Make solo cost prohibitive.** Goal that touches 10+ packages,
  none of which fit in one context window. Force the agent to
  delegate.
- **Make the brief command spawn directly.** Drop "use it" / "only
  if warranted" hedge language entirely. Replace with: "Step 1: call
  `bosun_spawn` for cloudflare. Step 2: call it for synology. Step
  3: ‚Ä¶" The brief is a script the agent reads top-to-bottom.
- **Test the bosun_spawn tool itself in isolation.** A unit-style
  trial: drop the agent into a worktree, brief says "your one and
  only job is to call `bosun_spawn` with these exact arguments and
  report what comes back." Validates the tool wiring before
  validating the decision-making.

## Findings worth filing

1. **`bosun_spawn` decision UX is the actual gap.** An agent will
   route around `bosun_spawn` when the work feels tractable solo,
   even when the brief directs otherwise. The tool's value
   proposition isn't visible to the agent at decision time. Two
   product directions: (a) make the brief preamble harder to
   override ("you MUST spawn if there are ‚â•3 disjoint packages"),
   or (b) reframe the tool's pitch ‚Äî bosun_spawn is for cases the
   PARENT'S context isn't sufficient to hold all the work, not for
   speed. Today's tool description and the brief's language both
   pitch it as "parallelism," which the agent rationalizes against.
2. **No visibility into "why didn't you spawn?" from the operator
   side.** If the agent considered and rejected spawn, the operator
   has no signal (no log line, no spawn-tree entry, no MCP audit).
   A v0.9.x followup could have `bosun_spawn` returns logged to
   `.bosun/audit/spawn.log` even on refusal ‚Äî and the brief
   preamble could ask the agent to write a one-line rationale when
   it decides solo over spawn for any work the brief flagged as
   spawn-candidate.
3. **Configured-vs-broken distinction is a real product idea.** The
   agent invented a "configured" flag in the `HealthResult` to
   distinguish unset env vars from runtime failures. Worth landing
   that pattern in bosun's own surface (e.g., `bosun status` could
   show "no Claude API key configured" distinctly from "Claude API
   call failed").
