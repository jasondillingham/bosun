## Trial cut short after init phase

The trial did not reach Phase 2 (multi-session round). Two real bugs
in the init path surfaced within 15 minutes; rather than push through
hand-recovery to exercise the merge/cleanup gates, the trial was
stopped here to surface fixes before re-running. **Important: the
v0.6 safety contract held throughout.** Main was untouched, no code
was corrupted, no work outside `<repo>` or `<repo>-bosun-*` was
written. Recovery was manual (`git worktree unlock` + `remove
--force`, `rm .bosun/init.state`, `git branch -D`) but clean.

## Findings

### Finding 1 — **`bosun init` hangs on `git worktree add` with no timeout**

**Severity:** high (blocks adoption on under-resourced systems).
**Affects:** v0.6 main.

`bosun init 3 --brief v0.6-trial-plan.md` printed:

```
system load is 7.26; init may be slow (--no-load-check to skip)
Creating worktree session-1 (1/3)...
```

…then hung for 14 minutes with zero output before being killed. The
v0.5 per-op git timeout (default 30s per `docs/v0.5-roadmap.md`) did
not fire. Audit needed: either `git worktree add` is not wrapped in
the timeout, the timeout default is much higher than 30s for this
operation, or the timeout fires but the command silently retries.

The filesystem side of the operation completed (the
`homelab-status-mcp-bosun-1/` worktree was populated with CLAUDE.md,
cmd/, go.mod, etc.), and `git worktree list` showed the worktree
registered but `locked`. So `git worktree add` itself returned to bosun
— bosun is hanging on something AFTER the worktree-add call, or the
git subprocess hangs on fsync after writing but before returning.

**Confounder:** system load was 6-11 during the run. Load contributed
but does not excuse missing timeouts — the entire reason the timeouts
exist is to bound exactly this case.

**Repro:**
1. Pre-load system to 1-min avg > 5 (e.g. `find / -type f >/dev/null &`).
2. `cd <empty-go-repo> && bosun init 3 --brief plan.md`
3. Observe hang after "Creating worktree session-1 (1/3)..." with no
   timeout fire.

**Fix shape (v0.6.1):**
- Verify `internal/git/Client.run` actually applies `context.WithTimeout`
  around the underlying `exec.CommandContext` for `worktree add`. Add
  test that wraps a fake git that sleeps 60s and asserts the call
  returns within timeout+5s.
- Default timeout for `worktree add` specifically may need to be
  longer (60-120s) since worktree creation under fsync pressure is
  legitimately slow, but the cap MUST exist.
- Error message on timeout: "git worktree add timed out after Xs —
  system likely under fsync pressure; check `uptime` and retry, or
  `bosun init --resume` to continue."

### Finding 2 — **`bosun init --resume` doesn't actually resume**

**Severity:** high (defeats the entire v0.6 ask #7).
**Affects:** v0.6 main.

After killing the hung init, `.bosun/init.state` correctly recorded:

```json
{
  "version": "v0.6",
  "current_session": "session-1",
  "current_step": "git_worktree_add",
  "completed_sessions": []
}
```

So far so good — the breadcrumb mechanism works. But the resume path
has two distinct failure modes:

**2a — `bosun init --resume` (no args) refuses with arg-count
mismatch:**

```
$ ~/go/bin/bosun init --resume
Resuming previous init (started 2026-05-17T02:19:52Z).
Error: bosun: --resume label count (4) doesn't match prior init (3).
       Re-run with the same args, or remove .bosun/init.state to start
       fresh.
```

Where does "4" come from? Nothing in my invocation suggests 4. Either
the default count is being injected before --resume is checked, or
there's an off-by-one. Either way, the operator expectation is that
`--resume` resumes from `init.state` — you shouldn't have to remember
your original args.

**2b — `bosun init 3 --resume --brief v0.6-trial-plan.md` refuses
because the half-built worktree exists:**

```
Resuming previous init (started 2026-05-17T02:19:52Z).
system load is 7.26; init may be slow (--no-load-check to skip)
Error: bosun: worktree path already exists:
       /Users/jasondillingham/Documents/Homelab/homelab-status-mcp-bosun-1
       (use --force to overwrite)
```

This is the wrong refusal. The whole point of init.state's
`current_step: git_worktree_add` is to remember that session-1's
worktree was being created when we died. Resume should:

1. See `current_step: git_worktree_add` for session-1.
2. Check whether the worktree already exists + is properly registered
   in `git worktree list`. (It is — `git worktree list` showed
   `homelab-status-mcp-bosun-1 ... [bosun/session-1] locked`.)
3. If yes → `git worktree unlock`, mark step complete, continue to
   `state_file_write`.
4. If partially populated but not registered → that's the case
   `--force` should handle.
5. Only fail if the operator passes inconsistent args (different brief,
   different count) AND --force is not set.

**Confounder:** the worktree was `locked` because bosun's kill left
the lock file. Resume should handle that — a lock from a prior
bosun-owned init is recoverable, not fatal.

**Fix shape (v0.6.1):**
- Make `--resume` actually use `init.state` as the source of truth for
  count + brief path. Operator should be able to run `bosun init
  --resume` from anywhere in the repo and have it pick up.
- Make resume detect "worktree exists, locked by bosun" as the
  expected case for the in-progress step. Unlock it, mark the step
  complete, advance.
- Surface `bosun init --status` (read-only) that reads init.state and
  prints what's done / what's pending without taking action — useful
  for operators trying to understand a wedged init.

### Finding 3 — pre-flight load check fires correctly (✅)

`system load is 7.26; init may be slow (--no-load-check to skip)` —
v0.5 feature working as designed. Warning, not refusal. Good
trade-off.

### Finding 4 — init progress reporting works (✅)

"Creating worktree session-1 (1/3)..." — v0.5 progress reporting
fires before each worktree-add. Without this, the hang in Finding 1
would have been "silent for 14 minutes" instead of "stuck on session
1 for 14 minutes." Material improvement.

### Finding 5 — `.bosun/init.state` breadcrumb mechanism works (✅)

v0.6 ask #7's persistence layer is solid. The JSON file got written
to disk with the right shape and survived the kill. The bug is in
the consumer (`--resume`), not the producer.

### Finding 6 — safety contract held under failure (✅✅✅)

This is the most important observation in the trial:

- Pre-trial main SHA: `667b7f3`.
- Post-recovery main SHA: `667b7f3`. **Unchanged.**
- No commits in the trial repo's history that bosun didn't make
  explicitly.
- No files in `~/Documents/Homelab/homelab-status-mcp/` modified by
  bosun outside `.bosun/` (which is bosun's documented territory).
- Worktree under `homelab-status-mcp-bosun-1/` was the only
  out-of-tree write, exactly per the safety contract.
- Recovery via `git worktree unlock`, `git worktree remove --force`,
  `rm .bosun/init.state`, `git branch -D bosun/session-1` was
  documented git operations — no bosun-internal magic required.

**This is exactly the contract the public-launch readme needs to
make.** Findings 1 and 2 are real bugs but they're operator-pain bugs,
not safety-contract violations. The user can recover from both with
documented git commands. That's the right shape.

## Verdict for the public launch

**Not yet.** Findings 1 and 2 are blockers — a Hacker News landing
user who pre-loads their system and watches bosun hang for 14
minutes will not have a good experience. Both are 1-2 day fixes per
v0.6.1.

**But the safety contract held.** That's the load-bearing piece. With
the init reliability bugs fixed, the v0.1 public launch is plausible
on a v0.6.1 → second trial → ship timeline.

## Addendum (2026-05-17) — v0.6.1 round confirmed the bug is broader

During the v0.6.1 dogfood round itself, `bosun merge session-1` hung
for 11 minutes on the pre-merge `git fsck --no-progress --no-dangling`
call (the v0.6 session-1 feature). 0.22 seconds of CPU time during that
window — pure I/O wait. Load was 10.29.

This expands Finding 1 from "`git worktree add` is missing a timeout"
to **"the per-op timeout doesn't wrap ANY git operation, including
the new fsck call."** The v0.6.1 session-1 fix in `internal/git/git.go`
addresses `worktree add` specifically; it needs to be audited to make
sure EVERY git invocation that goes through `internal/git/Client.run`
inherits the timeout — fsck, merge-squash, branch ops, rev-parse,
worktree list, all of them.

The trial repo remained safe (manual `git merge --squash` recovered),
but the v0.6 merge-fsck-then-merge flow is currently unusable under
load.

## v0.6.1 punchlist

1. **Audit per-op git timeouts.** Verify `git worktree add` is
   timeout-wrapped. Add a test with a fake-git shim that sleeps 60s
   to prove it. Bump default to 60-120s for worktree-add specifically
   if needed. **Update (2026-05-17):** the audit also needs to cover
   `git fsck` (the v0.6 pre-merge call) and any other op that goes
   through `internal/git/Client.run`. The trial-repo finding was
   `worktree add`; the dogfood-loop finding was `fsck`. The fix
   shape is the same — wrap the run helper itself, not individual
   call sites.
2. **Fix `bosun init --resume`** to read everything from init.state
   (count, brief, args) without requiring them on the command line.
3. **Make resume tolerate the locked-in-progress worktree case.**
   Unlock + advance, don't fail.
4. **Add `bosun init --status`** for read-only inspection of
   init.state.
5. **Document the safety-contract validation in README.** The third
   finding above is the headline; lift it verbatim into the safety
   contract section session-4 of v0.6 round-1 added.

## Trial protocol notes (for next round)

- The "wait for load < 5" pre-flight gate exists for a reason.
  Operator-override is fine but document the resulting findings as
  load-confounded.
- Capture pre-trial HEAD SHA before any bosun command — this trial's
  cleanest evidence was "HEAD unchanged after recovery."
- Phase 2 (multi-session round) needs a clean Phase 1 first. Init
  reliability is the gating dependency.
- The v0.6 predictor flagged 79 false-positive overlaps when run
  against the v0.6 round 1 plan because it treats "Files (avoid)"
  sections as predicted edits. Worth fixing before this trial repeats
  (different bug, different round).
