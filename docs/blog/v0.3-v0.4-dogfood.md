# Dogfooding bosun on bosun: v0.3 + v0.4 field notes

*Posted 2026-05-16*

Bosun is a Go CLI for coordinating parallel Claude Code sessions on isolated git worktrees. The fastest way to find out where it leaks is to use it to build itself. v0.2 shipped the MCP protocol surface; v0.3 and v0.4 are the rounds where bosun started being load-bearing for its own development. Here's what landed, what didn't, and the bugs the dogfood loop caught that the test harness missed.

## What landed in v0.3

The operator-facing surface. v0.2 was for the agents — six MCP tools so a session can `bosun_claim` / `bosun_release` / `bosun_done` / `bosun_stuck` / `bosun_announce` / `bosun_check` directly through tool calls. v0.3 was for the human running the fleet:

- **`bosun tui`** — a long-running Bubbletea control center. j/k to move, `m` to merge the selected session, `M` to merge every DONE, `c` to cleanup, `r` to remove, `l` to launch. Refreshes every 2s.
- **`bosun serve`** — an HTTP dashboard with `/api/status` (JSON) and `/api/events` (SSE). One browser tab beats six terminals.
- **Custom session labels.** `bosun init auth http storage` works alongside `bosun init 4`. Plan-markdown briefs can target `## auth` instead of `## session-1`.
- **Agent process detection** in `internal/proc`. Status now has a RUNNING column that's accurate within "is there a `claude` / `claude-code` / `code-cli` process whose CWD points at this worktree?"

Plus v0.3.1, a bug-hunt round that surfaced two real coordination bugs (covered below).

## What landed in v0.4

The integration-rehearsal round. v0.4 is what the operator reaches for when they want to *check* a merge before doing it, or wire bosun into a bigger system.

- **Lifecycle hooks scaffolding.** `pre-init` / `post-init` / `post-done` events, wired through `.bosun/config.json`, run through `sh -c`. `fail_open` controls whether non-zero exits block or just warn. `pre-merge` / `post-merge` / `pre-cleanup` / `post-cleanup` / `pre-remove` are the v0.5 ask — scaffolding only in v0.4.
- **`bosun merge --dry-run`.** Print what would merge, in dependency order, without touching the work tree. Pairs with the new `--json` flags for scripted preflight.
- **`bosun list --json` / `bosun show --json`.** Stable wire shape (`version: "v1"`). External dashboards and CI gates no longer have to parse human-readable tables.
- **Web brief preview + announcement feed.** `bosun serve` now renders BOSUN_BRIEF.md and surfaces `bosun_announce` events on the SSE stream.
- **`bosun config`** — inspect and edit `.bosun/config.json` without hand-editing JSON.
- **Orphan-directory recovery.** `bosun cleanup --orphans` now reconciles in both directions — stale git registrations *and* on-disk worktrees with no git registration.
- **Bug-hunt audit** on `internal/claims/` / `internal/state/`, motivated by the v0.3.1 flock race.

## What didn't land

The v0.4 brief is honest about this: pre/post-merge, pre/post-cleanup, pre-remove hook call-sites are queued for v0.5. The hooks runner is real; the call-sites are the v0.5 ask. The release is tagged `-rc1` partly to signal that.

Predictive conflict analysis (LSP-driven, full call-graph) also continues to slip — the v0.5 roadmap proposes a first slice using filename heuristics on plan markdown, but the heavyweight version isn't on any near-term map.

Ghostty tab support: dropped in v0.2 polish because the Ghostty CLI doesn't expose `+new-tab`. Still pending upstream; v0.5 will rewire it if Ghostty ships, otherwise it slips again.

## The bugs the bug-hunts found

The v0.3.1 bug-hunt round was the most valuable single hour of v0.3.

### Cross-process claims race (commit `333ff36`)

Symptom: two writers updating `.bosun/claims/<session>.json` would silently lose whichever update finished second. Easy to miss because each writer's read-modify-write looked atomic from the inside.

Cause: bosun was using in-process locks. When a CLI invocation and the MCP server both wrote claims simultaneously, neither was aware of the other's read-then-write window.

Fix: `flock(2)` on the claims file. Cross-process advisory locking, ten lines of platform code. The boundary tests written for the bug-hunt round caught the race directly — they spawn two processes that race each other into the same `bosun_claim`, then assert that both updates survive.

The lesson: in-process tests don't catch cross-process bugs. The MCP server introduced a *second* process that touches the same files the CLI touches, and the test harness needed to grow to match.

### Stale `BOSUN_MCP_SOCK` discovery wedge (v0.3.1)

Symptom: a session would silently fail to connect to the MCP server. `bosun_check` calls would return "no overlaps" no matter what other sessions claimed. The operator's shell had a `BOSUN_MCP_SOCK` from a previous bosun-managed repo still exported, and child sessions inherited it.

Fix: `inheritedSocketBelongsToRepo` check. The launcher confirms the inherited socket path is rooted under the current repo's `.bosun/mcp.sock` before honoring it; otherwise it ignores the inherited value and falls back to the local socket.

### Read-only GOMODCACHE on `bosun remove --force`

Symptom: `bosun remove --force` failed with `EACCES` on Go module cache contents inside a worktree.

Fix (v0.3.1, partial): `internal/git.chmodWritableTree` walks the worktree and chmod's read-only files writable before invoking `git worktree remove`. A test was written using a realistic-shape GOMODCACHE tree — and it passes — but the production failure on `bosun-bosun-1` still reproduced. The test doesn't yet capture the actual failure mode. Captured in `docs/v0.4-findings.md` for a future round.

## The silent-hang lesson from v0.4 kickoff

v0.4 round 1 kickoff started with `bosun init 10 --brief v0.4-plan.md` and went silent. Not failed — silent. `ps` showed `git worktree add` running at 0.02s CPU, hung on what looked like an APFS fsync stall behind whatever else the kernel was flushing (Spotlight reindex + iCloud verification kicked off by an unrelated cleanup task).

bosun didn't catch this gracefully:

1. No timeout on any `git` subprocess invocation in `internal/git/git.go`. The hang continued until the operator killed bosun.
2. No progress output between "starting init" and "created N sessions." From the operator's seat, hung-on-fsync looked identical to "any moment now."
3. No recovery path. The worktree was half-created on disk; the next `bosun init` didn't reuse it.

Three different gaps. The v0.5 roadmap proposes per-operation timeouts (default 30s, configurable), progress reporting on every `AddWorktree` call, and a pre-flight load-average check. All three small fixes, all three needed independently.

The meta-lesson: a tool that wraps subprocesses is only as good as its weakest subprocess invocation. bosun's `git worktree add` is fine 99% of the time; the 1% needed a timeout from day one. Captured in `docs/v0.4-findings.md`.

## What the dogfood loop is actually for

This isn't a brag piece — the bugs above shipped because bosun was *used* on real work, not because the test harness found them. The harness caught what we knew to test for. The dogfood loop caught everything else: concurrent-writer races between two cooperating bosun processes, stale-env wedges across operator shells, fsync hangs under macOS background load, read-only Go cache files surviving force-remove.

If you're writing a tool whose whole purpose is enabling parallel work on a codebase, the most honest test environment is to use it on the codebase you're writing it on. Every round of bosun development found at least one bug that nobody would have written a test for in advance.

v0.5 will follow the same shape: ship the planned scope, then dogfood it on v0.6 to find out what we didn't think to test.
