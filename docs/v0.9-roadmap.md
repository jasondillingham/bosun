# Bosun v0.9 — Roadmap

**Baseline:** v0.8.0 (public-launch readiness; trial #2 cleared the safety-contract gate, doctor + --fix + init --suggest shipped, repo flipped public).
**Anchor:** **Agent-driven coordination.** v0.1-v0.8 made bosun safe to point at a stranger's repo. v0.9 makes the agent itself an active coordinator — a session can spawn sub-sessions under the operator's authorization, fanning out work the agent itself decides to parallelize.
**Status:** Planning. Drafted after v0.8 shipped (2026-05-17 evening).

---

## Position relative to the prior roadmaps

The v0.6 roadmap explicitly deferred multi-machine / distributed coordination to "v1.0+ research project." This is **not** that — it's still single-machine — but it does open the agent-as-coordinator door the prior roadmaps kept closed.

This is the first round where bosun delegates *initiative* to the agent. Up to v0.8, every action that creates a branch / worktree / new session originates from a human typing `bosun init`. v0.9 introduces a path where a Claude session inside a worktree can call an MCP tool that creates more sessions — still under the operator's authorization (the operator flips a config gate first), but the *trigger* moves from the operator's keyboard to the agent's reasoning.

That's a real architectural shift. Get the safety + auth model right before any code.

---

## Why agent-driven, why now

Three drivers:

1. **The v0.7 round-1 dogfood showed the bottleneck.** Four parallel agents shipped 1525 lines in 25 minutes — a 3-4× speedup over a single Claude session. The bottleneck wasn't the work; it was the operator-side coordination (writing the brief, partitioning lanes, watching status, deciding when to merge). The agent has more context to write the brief than the operator does, especially for sub-tasks the agent discovers mid-work.

2. **The safety contract is empirically validated.** v0.8 trial #2 ran agents through SIGBUS, CRASHED, corrupted-gitdir, and merge-undo. Main stayed at its baseline SHA across every test. That's the trust foundation for letting agents trigger more agents — if a sub-agent goes off the rails, the contract still holds.

3. **Real users will hit the "I wish my agent could fan this out itself" wall.** The public launch surfaces real workflows. An agent partway through a refactor that realizes "this work would parallelize cleanly across three modules" is a common shape. Currently the only path is for the agent to stop, ask the operator, wait for a new bosun init. That breaks flow.

## The v0.9 asks

Five asks. Lane 1 ships the spec (no code); lanes 2-4 can run in parallel after.

### 1. `docs/v0.9-spawn-spec.md` — the design document

Before any code, settle the design questions that ripple through everything:

- **Authorization tiers.** Repo-level opt-in via `.bosun/config.json`. Default off. What config keys, what defaults?
- **Spawn quotas.** Per-session max concurrent sub-sessions (default 3). Per-tree max depth (default 1 — parents can spawn, grandchildren can't). How is depth tracked?
- **Branch model.** Do sub-sessions branch from parent (hierarchical merge) or from main directly (flat)?
- **Label scheme.** Parent `session-1` spawns a sub — what's its label? `session-1-auth`? `session-1.1`? Must remain valid under `session.ValidateLabel`.
- **Brief contract.** Agent provides the brief inline (markdown string) or via path? Validated the same way `bosun init --brief` validates.
- **Tool surface.** Single `bosun_spawn` tool taking `(parent_label, brief)` and returning the created sub-labels, or a tighter shape?
- **Recovery.** Parent dies — sub-agents keep working or get adopted by operator. Sub-agent dies — does parent get a signal? How?
- **Operator visibility.** TUI/status tree view, events log entry on spawn, optional per-spawn confirmation mode.
- **Termination.** Operator kills the parent — sub-agents inherit? Cleanup with `bosun cleanup --tree session-1` blows away the whole subtree.

Spec lands; lanes 2-4 implement against it.

### 2. `bosun_spawn` MCP tool + auth scaffolding

The tool itself:

- New file `internal/mcp/tool_spawn.go` implementing the spawn MCP tool.
- Auth gate: read `.bosun/config.json` to verify `agent_spawn.enabled` is true.
- Quota gate: read the spawn tree file to verify the parent hasn't exceeded `max_concurrent_sub_sessions`, and the parent's depth is under `max_depth`.
- Brief parsing: reuse `internal/brief.Parse` (or a string variant) to validate the agent's brief markdown.
- Spawn execution: reuse the bulk of `cmd_init.go`'s per-session creation logic, refactored so the MCP tool can call it without duplicating the per-session pipeline.
- Spawn tree tracking: new `.bosun/spawn-tree.json` records parent-child relationships, depth, timestamps. Used by quotas and by the status tree view.
- Event emission: write a `session_spawned` event to `.bosun/events.jsonl` so TUI / web UI surface it live.

### 3. Hierarchical status + visibility

The operator needs to see the spawn tree at a glance:

- `bosun status --tree` (or always-on when spawns exist) renders a parent-child indentation:
  ```
  SESSION        BRANCH                  STATE    AHEAD  DIRTY  ...
  session-1      bosun/session-1         WORKING  0      2      ...
    └─ session-1.auth   bosun/session-1.auth   DONE     1      0
    └─ session-1.http   bosun/session-1.http   WORKING  0      1
  session-2      bosun/session-2         DONE     1      0      ...
  ```
- TUI (control center) groups sub-sessions under their parent.
- Web dashboard shows the tree on `/`.
- `bosun show <session>` for a parent lists its sub-sessions; for a sub-session, names its parent.

### 4. Recovery + cleanup tooling

Closing the operator-control loop:

- `bosun cleanup --tree <session>` reaps a parent and all its descendants. Per-leaf safety: each sub-session goes through the same liveness/dirty gate `bosun cleanup` already enforces.
- `bosun adopt <sub-session>` flattens a sub into a top-level session. Useful when the operator decides "the parent's done, but I want to keep working on this sub independently." Removes the parent reference from spawn-tree.json.
- `bosun remove --tree <session>` per-tree analog of `bosun remove --force`. Single confirmation, then cascade.
- TUI key bindings: `t` to toggle tree view, `T` to collapse all parents.

### 5. Tests + trial #3

Same shape as v0.7 and v0.8 closing rounds:

- Unit tests for the auth gate, quota gate, depth tracking, label validation.
- Scenarios test that exercises a real spawn end-to-end (parent inits, calls bosun_spawn via the MCP socket, sub-sessions land on disk, status renders the tree correctly).
- Fuzz tests for the brief parser variant that takes inline strings.
- External-repo trial #3 against a real repo with a 2-level spawn tree (parent + 2 sub-sessions). Validates the safety contract under the new auth model.

---

## What's explicitly NOT in v0.9

- **Sub-agents calling `bosun_merge` or `bosun_remove`.** Destructive operations stay operator-only. Agents can spawn but can't reap.
- **Cross-machine spawning.** Single-machine assumption stays from the v0.6 roadmap.
- **Unbounded depth.** `max_depth: 1` is the default and we don't ship a way to lift it without source modification in v0.9. Operators who genuinely want deeper trees can edit the constant and rebuild; production callers shouldn't.
- **Auto-merging spawned subs back to parent.** Operator still drives merges. The hierarchy is for visibility + cleanup, not for autopilot.
- **Spawn-budget billing / cost surfacing.** Operators eyeballing token spend is fine for v0.9; structured cost tracking is a v1.x problem.
- **Sub-session-specific hooks.** Spawn-time hooks (pre-spawn / post-spawn) might be useful but defer until real users ask.

## Sequence

Five-lane round, but lane 1 must land before 2-4 can run in parallel.

1. **Lane 1 — `docs/v0.9-spawn-spec.md`.** No code. Maintainer-driven design doc. Ships first; subsequent lanes implement against it.
2. **Lane 2 — `bosun_spawn` MCP tool + auth/quota/tree-tracking.** Touches `internal/mcp/`, `internal/config/` (new `agent_spawn` section), new `internal/spawntree/` package, possibly small surgery on `cmd/bosun/cmd_init.go` to extract the per-session create helper.
3. **Lane 3 — Hierarchical status + tree view.** `internal/session/derive.go`, `cmd/bosun/cmd_status.go`, `cmd/bosun/cmd_show.go`, `internal/tui/`, `internal/web/handlers.go`.
4. **Lane 4 — Recovery + cleanup tooling.** `cmd/bosun/cmd_cleanup.go`, new `cmd/bosun/cmd_adopt.go`, `cmd/bosun/cmd_remove.go`.
5. **Lane 5 — Tests + external trial #3.** Unit + scenario tests across the above; trial doc.

Lanes 2-4 are parallel-friendly by file once lane 1 settles the design. Lane 5 runs after they all land.

---

## After v0.9

- **Re-trial against an external repo with agent-driven spawning.** If trial #3 stays clean, the trust story extends to "agents can fan out under operator authorization without breaking the safety contract."
- **v1.0.** Reasonable milestone once v0.9 has a few weeks of public use. Consolidates v0.7/v0.8/v0.9 into a single-digit-versioned story instead of cumulative 0.x. The v1.0 anchor should not be a new feature — it should be the confidence interval. If v0.7-0.9 have shipped with no safety-contract violations reported by public users, v1.0 is just the milestone.

## Open questions

1. **Branch hierarchy or flat?** If sub-session-1.auth branches from session-1, merging session-1 to main requires merging the sub first. That's a hierarchy. If sub branches from main, merges are independent but the "parent's work depends on sub's output" relationship is lost. Strong opinion needed in the spec.

2. **What's the agent's view of its sub-sessions?** Today MCP tools work over the bosun socket from any worktree. A parent agent could call `bosun_status` and see its own subs. Is that enough or do we need a `bosun_subsessions` tool that only returns the calling agent's own subs?

3. **Quotas: per-session or per-tree?** Default `max_concurrent_sub_sessions: 3` per parent. Tree-wide quota is harder to reason about (different sub-trees compete for budget) but easier to bound resource use. Lean toward per-session for the simpler mental model.

4. **What does sub-failure look like?** A sub-session fails its `make check`. Today the operator sees it via `bosun status`. With agent-driven coordination, should the parent agent be notified (heartbeat-style)? Or does the parent just poll? Bias toward poll (simpler; lets the agent decide its own cadence).

5. **--depth=N escape hatch?** Some operators will genuinely want a 2-level tree. Default `max_depth: 1` but operator can config-override to a higher value. Bound to some absolute ceiling (say 4) so a misbehaving agent can't fork-bomb. Worth discussing in spec.

6. **Trial-#3 target repo.** Same homelab-status-mcp as trials 1+2? Or pick a fresh repo? Bias toward the same repo for continuity; a fresh repo doesn't add safety signal since the contract isn't repo-specific.
