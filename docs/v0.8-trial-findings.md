**Behavior is safe** (merge refused; main untouched) but the message is the simpler v0.5-era "uncommitted changes" check, not the v0.6 brief's intended:

> `bosun: session-N has a live agent (pid 12345) and uncommitted changes — refusing merge`
> `       to avoid dropping in-flight work. Recovery:`

## Findings

### Procedural / environmental (outside bosun's control, but documented for operators)

1. **Trial repo recovery required manual git surgery.** A prior aborted run left the repo with a corrupted `internal/parsers/testdata/uptime_typical.txt` and a stale `.git/index.lock`. `bosun doctor` caught some of the leftover state (stale `.bosun/init.lock`) but not the git-level corruption. Recovery: `rm` the corrupted file → `rm .git/index.lock` → `git checkout HEAD -- <file>`. Operator can dispatch this from doctor's output but doctor doesn't autofix.

2. **`mv` across volumes is `cp` + `rm`.** Moving a 15M repo across volumes via `mv` invoked `fcopyfile` for each git object; multiple files hit "Operation timed out" partway through. Caused by iCloud's File Provider concurrently virtualizing the source files (`ditto` also failed with phantom "No such file or directory" errors on files that demonstrably existed). **The reliable approach was `tar | tar` piping** — single I/O stream, no per-file fcopyfile, transparent to iCloud's lazy-hydration.

3. **macOS Sequoia ships `openrsync`, not GNU rsync.** Flags like `--info=progress2` are silently rejected (rsync prints its usage and exits 0). `--partial` works but the progress UX is absent. Worth flagging in any doc that recommends rsync.

4. **Dock-attached external M.2 drives are slow under load.** The drive itself is APFS/journaled (fine), but the USB controller in the dock dropped reads under concurrent I/O. Single-stream copy (`tar | tar`) worked; many-small-file copy (`ditto`) didn't. The user offered to plug the drive direct to MacBook — worth doing for the trial run itself, not just the copy.

### Bosun-side findings

5. **Safety contract held under SIGBUS.** Attempt 1's init failed mid-`git worktree add` checkout. Bosun emitted a clear error naming the operation + exit code 138, left the trial repo's HEAD unchanged, did not create any half-state worktree dirs, and wrote an `init.state` breadcrumb. `bosun init --resume` correctly picked up the breadcrumb on retry. **This is the load-bearing validation for the public-launch gate** — when bosun fails because of an environmental hazard, the failure is clean.

6. **Doctor's iCloud-sync warning is load-bearing operator guidance.** On attempt 1, doctor identified the exact failure cause before init even ran. The recommendation ("move the repository to ~/code/ or similar") was correct. **Doctor's value is fully validated.**

7. **`bosun init --suggest` not yet exercised in trial** — the brief was hand-written. Worth a future trial that exercises the `--suggest` path against a fresh external repo.
