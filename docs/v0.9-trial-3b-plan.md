# Trial #3b — spawn-warranted brief

Follow-up to [`v0.9-trial-findings.md`](./v0.9-trial-findings.md).
Trial #3a's brief left spawn discretionary; the audit found only one
package needed work, so the agent did it solo. Correct call, but
`bosun_spawn` never fired. **Trial #3b's job is to land a brief whose
work is genuinely impossible-to-shortcut parallel across three
packages**, so the parent must spawn for the round to finish in
reasonable time.

## Design principles

A brief is "spawn-warranted" when it satisfies all three:

1. **N disjoint package-shaped lanes** — each lane edits files in a
   single package the others don't touch. No shared types or config.
2. **Each lane is substantial** — at least 30-60 lines of real work per
   package. A one-line fix per package would still feel solo-doable.
3. **Genuine parent-side integration** — the parent isn't just "wait
   for subs"; it has a real piece of work that *depends on* the subs
   having landed. This is what exercises `bosun merge --tree`'s post-
   order cascade (subs merge into parent's branch first; then parent
   merges to main).

The original brief failed (1) ambiguously, (2) clearly (one-paragraph
doc per package), and (3) entirely.

## Target shape: cross-cutting feature

The strongest spawn-warrant shape is **"add the same capability to N
packages, then expose them through one new integration point"**:

- **Per-package work (spawn candidates):** add a `Health()` method to
  each of `internal/cloudflare`, `internal/synology`,
  `internal/tailscale`. Each must:
  - Have a typed return (`HealthResult struct { OK bool; Latency
    time.Duration; Detail string; Err error }`)
  - Hit one cheap read-only endpoint of the underlying service
  - Be fully unit-tested via httptest (no real network)
  - Be ~40-80 lines of new code per package
- **Integration piece (parent's work):** wire a new MCP tool
  `homelab_health` in `internal/mcp/` that calls all three `Health()`
  methods in parallel via `errgroup`, aggregates results into a single
  JSON response, and registers itself with the MCP server. ~80-120
  lines including tests.

Why this works:

- Each `Health()` lives in its own package — three disjoint worktrees,
  zero merge conflicts between subs.
- The MCP tool *cannot* be written before the three `Health()` methods
  exist — the parent has a real reason to wait for `merge --tree` to
  cascade subs first.
- Each piece is big enough that a single agent doing it solo would be
  noticeably slow.
- The parent-side work needs architectural decisions (error shape,
  parallel-call cancellation semantics, JSON schema) — exactly the
  kind of thing the parent should own rather than delegate.

## The brief (drop verbatim into `trial3b-plan.md`)

```markdown
# Trial #3b — spawn-warranted parallel feature add

## session-1

**Goal.** Add a uniform `Health()` capability to three service-client
packages, then expose them through one new aggregating MCP tool. The
three per-package implementations are genuinely independent; the
aggregating tool *requires* all three to exist before it can be
written.

### Architecture (your call to make and own)

- **Per-package interface** (you define this; document in your final
  message what shape you chose):

  ```go
  type HealthResult struct {
      OK      bool
      Latency time.Duration
      Detail  string
      Err     error
  }

  // Each of (*Client).Health(ctx context.Context) HealthResult
  // — exact signature is yours to fix, but the THREE packages must
  // agree.
  ```

- **Cross-cutting MCP tool:** `homelab_health` registered in
  `internal/mcp/`. Takes no arguments. Returns JSON with one entry per
  service. Calls the three `Health()` methods concurrently via
  `errgroup`. Aggregates results — a failure in one service does not
  fail the tool; it's reflected in that service's `ok: false`.

### Spawn instruction (read carefully)

You have access to the `bosun_spawn` MCP tool. **Use it.** The three
`Health()` implementations are disjoint package-shaped work — this is
exactly the shape spawn was built for. Suggested plan:

1. Read the three packages' existing `client.go` to understand their
   shapes. Pick the cheap read-only endpoint each will hit.
2. Decide the `HealthResult` shape (above) and write it down in this
   conversation so the subs all agree.
3. Call `bosun_spawn` three times:
   - Sub 1 (`session-1.cloudflare`): implement `cloudflare.Client.Health`
   - Sub 2 (`session-1.synology`): implement `synology.Client.Health`
   - Sub 3 (`session-1.tailscale`): implement `tailscale.Client.Health`
   Each sub gets a one-paragraph brief naming the target package, the
   chosen endpoint, the agreed `HealthResult` shape, and the tests
   expected.
4. While subs work, you write `internal/mcp/homelab_health.go`
   *against your own definition of the interface* — it doesn't matter
   that the implementations aren't in yet; the tool's code only needs
   the type to compile.
5. When all three subs are DONE, run `bosun merge --tree session-1`.
   This walks the spawn tree post-order: each sub squash-merges into
   your branch, then your branch (which has the MCP tool) is ready
   for the operator's final merge to main.
6. Mark yourself DONE.

### Constraints

- Each sub's per-package work touches one file in its named package +
  one test file. No cross-package edits inside a sub.
- All three subs must produce passing `go test ./internal/<pkg>/...`.
- Your MCP tool registration must produce passing
  `go test ./internal/mcp/...`.
- `go vet ./...` must pass on every branch.
- Don't touch `internal/audit`, `internal/ssh`, `internal/sslcheck`,
  `internal/parsers`, `internal/config`, `internal/uptimekuma`, or
  `internal/homeassistant`. They're not in scope for this trial.

### Done criteria

- Three `Health()` methods landed on your branch via spawn-tree merge.
- `homelab_health` MCP tool registered and tested.
- `bosun merge --tree session-1` cascades cleanly with all subs.
- Your final `bosun done` message names the chosen endpoints (one
  per service) and the `HealthResult` shape you settled on.
```

## What to watch during the trial

These are the v0.9 surfaces the trial is *specifically* designed to
exercise — capture each in the findings doc:

1. **`bosun_spawn` MCP call from parent.** Should succeed three times,
   each producing a sub-session entry in `.bosun/spawn-tree.json` and
   a worktree at `<repo>-bosun-1.<sub>` (or whatever the suffix
   pattern resolves to for dotted labels).

2. **Hierarchical session labels.** `bosun status` should render:

   ```
   session-1            WORKING ...
   ├─ session-1.cloudflare  WORKING ...
   ├─ session-1.synology    WORKING ...
   └─ session-1.tailscale   WORKING ...
   ```

   Same shape in the TUI and `/api/status` JSON (the v0.9.1 #6 tree
   rendering landed in this round).

3. **Quota refusal.** With `max_concurrent_sub_sessions=3`, the fourth
   `bosun_spawn` call should refuse with a clear quota message. The
   trial brief only asks for three subs so this won't fire organically
   — operator can force it post-trial as a sanity check.

4. **Depth refusal.** If a sub agent (curious or buggy) tries to spawn
   its own sub, it must refuse with the depth-1 message. Same caveat
   — won't fire organically; force it manually with a deliberate
   `bosun_spawn` call from inside one of the subs.

5. **`bosun cleanup` + spawn-tree refusal.** While subs are live,
   `bosun cleanup session-1` should refuse with "session-1 has live
   sub-sessions; cleanup the children first or use --tree".
   `bosun cleanup --tree session-1` should cascade.

6. **`bosun merge --tree` post-order.** The subs merge into
   `bosun/session-1` first; then `bosun/session-1` is ready to merge
   to main. Watch for "already merged" no-op detection when the
   parent merge happens (the subs' patches are already on the parent
   branch, so the patch-id matcher should see them and skip-not-fail).

7. **Safety contract.** Same checkpoints as trial #3a — capture `main`
   rev before and after every operation. Main only moves on operator
   `bosun merge`.

## How to drive the trial

This is where trial #3a couldn't be a true test: `bosun_spawn` lives
inside the MCP server and is callable only by a Claude Code session
that has `BOSUN_MCP_SOCK` in its env. The parent Claude session that
ran #3a wasn't bound to that socket.

For #3b to be a real test of `bosun_spawn`, the parent agent must be
launched via:

```sh
cd /Volumes/SneakerNet/bosun-trial/homelab-status-mcp
~/go/bin/bosun launch session-1
```

…in a fresh Ghostty window. That command sets `BOSUN_MCP_SOCK` on
the launched process so the agent's MCP client sees `bosun_spawn`
alongside `bosun_claim` / `bosun_done` / `bosun_check`. From there,
the agent reads `BOSUN_BRIEF.md` (which `bosun init --brief
trial3b-plan.md` writes into the worktree), and proceeds.

A parent Claude session driving via `Bash` + `Edit` tools — the way
#3a ran — *cannot* exercise this path. That's not a tooling gap; it's
the contract: `bosun_spawn` is opt-in for agents that have explicitly
been wired into bosun's MCP server.
