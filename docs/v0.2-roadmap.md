# Bosun v0.2 — Roadmap

**Baseline:** v0.1.0 (shipped 2026-05-15)
**Target shape:** v0.1.0's CLI plus an MCP server interface that lifts session coordination out of filesystem files into structured tool calls. The Bubbletea TUI control center stays deferred to v0.3 so v0.2 can ship the protocol work cleanly first.
**Status:** Planning. Nothing implemented yet; this doc is the agreement before work starts.

---

## The v0.2 anchor

### MCP server (`bosun mcp`)

A Model Context Protocol server that exposes bosun's session coordination as tool calls, so sessions can `bosun_claim` / `bosun_release` / `bosun_done` / `bosun_check` directly from inside an MCP-capable agent.

**Why:** Filesystem claims are advisory and require the agent to shell out to `bosun claim`. Tool calls make coordination first-class, lower-friction, and synchronous (with optional locking semantics).

**Tool surface:**
- `bosun_claim(paths: string[])` — declare paths the current session is editing
- `bosun_release(paths: string[])` — explicit release (currently happens at merge time)
- `bosun_done(message?: string)` — replaces touching `.bosun/state/<session>.done`
- `bosun_stuck(message: string)` — replaces touching `.bosun/state/<session>.stuck`
- `bosun_check(paths: string[]) → { conflicts: [{path, sessions}] }` — query whether any of these paths are claimed by another session before starting work
- `bosun_announce(event: string)` — broadcast something to the operator (surfaces in TUI)

**Where it runs:** `bosun mcp` starts an MCP server bound to a Unix socket at `.bosun/mcp.sock`. Sessions connect via the socket (configured in each worktree's `.claude/mcp.json` or equivalent). One server per bosun-managed repo.

**Compatibility:** Filesystem claims/state remain readable by `bosun status`, so a session not connected to MCP still works in v0.2. MCP is the preferred path; filesystem is the fallback.

**Dependencies added:** an MCP server library (TBD — likely `github.com/modelcontextprotocol/go-sdk` or roll thin custom JSON-RPC if the SDK is heavy).

## Smaller v0.2 work (post-MCP)

These are all small enough to land alongside or after the MCP server:

- **Tab support for non-Ghostty terminals.** iTerm2 has its own scripting (`tell application "iTerm2" to ... in current window`). Terminal.app similar. gnome-terminal has `--tab`. Windows Terminal has `wt new-tab`. Each is small but real platform work.
- **Orphan-session cleanup.** When `bosun init --force` goes from N=4 to N=3, session-4's worktree lingers. Add `bosun cleanup --orphans` to remove any bosun-managed worktree beyond the configured count.
- **`bosun launch <session>` standalone command.** Spawn a session-window without going through `init`. Useful for re-opening a closed window or testing the launcher.
- **`bosun status --summary-only` flag.** Just the one-line header, no table — for scripting and tiny terminals.
- **Custom session names.** Allow `bosun init auth http storage` (named) in addition to `bosun init 3` (numbered).
- **Dependency-aware plan briefs.** Plan markdown can declare `## session-2 (depends: session-1)` and bosun's merge respects ordering.

---

## Not in v0.2 (deferred to v0.3+)

- **Bubbletea TUI control center.** Persistent operator dashboard with live state, keybinds for actions (merge / cleanup / remove / launch), brief preview pane. Originally proposed for v0.2 but parked back at v0.3 so the MCP work lands first without competing for attention.
- **Web dashboard.** Same data as the TUI but over HTTP+SSE. Reuses the existing job-hunt dashboard's shape.
- **Hooks** (pre-init, post-merge, etc.) — v0.4 unless someone hits a specific need
- **Predictive conflict analysis** (knowing what a session *will* touch before it touches it) — needs static analysis or LSP integration; v0.3+
- **Auto-detection of "is an agent process actually running in this worktree?"** — adds a column to status / a TUI indicator
- **Resource governance beyond `--isolate-cache`** — scheduling test runs across sessions, etc.

---

## Sequence

Suggested order:

1. **MCP server v1** — bare protocol: `bosun_claim`, `bosun_done`, `bosun_check`. Filesystem fallback stays. Dogfood it on the next round of bosun work.
2. **MCP server v2** — locking semantics for `bosun_claim` (synchronous "is this path safe?" rejection), `bosun_announce` for operator-visible events
3. **Smaller v0.2 polish** — terminal tab support for non-Ghostty, orphan cleanup, named sessions, etc., as friction surfaces between MCP iterations

Each step is a dogfood round target.

---

## Open questions

1. Does the MCP server share state with filesystem claims, or replace them entirely behind a feature flag?
2. Does `bosun mcp` need authentication for the socket, or is filesystem permission enough?
3. Do we ship our own MCP server library, or take a dependency on an existing one (e.g. an official Go SDK)?
4. How does the server lifecycle work — is `bosun mcp` a foreground daemon the operator starts, or do bosun init / status auto-start it as a sidecar?
