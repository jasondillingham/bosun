# v0.9 trial #3c findings — homelab-status-mcp (2026-05-17)

## Outcome

**Recipe-style brief works.** The parent agent obeyed the command-
mode framing and called `bosun_spawn` three times for the per-
package lanes. Trial #3b's "agent prefers solo when work feels
tractable" failure mode did not reproduce.

**But:** the v0.9 spawn lifecycle has real bugs downstream of the
spawn call. The three sub-worktrees were created, then disappeared
mid-trial under unclear circumstances. The parent never got a
signal that its subs had vanished and continued the work solo
(violating the brief's "do not write per-package code" constraint).
The trial exposed at least four distinct bugs in the v0.9 surface
worth investigating in a focused bug-hunt round.

**Safety contract held.** Main pre-trial = post-trial = `5117d69`.

## What was supposed to happen (recipe steps)

1. Write `internal/healthproto/healthproto.go` + commit
2. `bosun_spawn` session-1.cloudflare
3. `bosun_spawn` session-1.synology
4. `bosun_spawn` session-1.tailscale
5. While subs work, write `internal/mcp/tools_health.go` etc.
6. `bosun merge --tree session-1` to cascade
7. `bosun done session-1`

## What actually happened

1. ✓ Parent wrote `internal/healthproto/healthproto.go` (`4397a4d`)
2. ✓ `bosun_spawn` session-1.cloudflare at 03:20:58Z — worktree
   `homelab-status-mcp-bosun-1.cloudflare` created, branch
   `bosun/session-1.cloudflare` registered, spawn-tree entry written
3. ✓ `bosun_spawn` session-1.synology at 03:21:04Z — same
4. ✓ `bosun_spawn` session-1.tailscale at 03:21:06Z — same
5. ❌ **Between 22:21 and 22:34 local time (~13 minutes), all three
   sub-worktrees disappeared from disk, all three sub-branches were
   deleted, and any sub-agent processes vanished from the process
   table.** spawn-tree.json was NOT updated to reflect the removal —
   the entries persisted as ghost references.
6. ❌ Parent did not call `bosun merge --tree` (step 6 of the recipe
   skipped entirely)
7. ✓ Parent committed all the per-package Health() work + the
   integration tool in one big solo commit (`0a37bd5`, 12 files,
   +880 lines) — violating the "do not write per-package code"
   constraint
8. ✓ Parent called `bosun done session-1` with 13 paths claimed

## Bugs surfaced

### Bug A — spawn-tree state can diverge from git's worktree state

**Severity:** Medium. State inconsistency that downstream commands
(merge --tree, cleanup --tree, show) all read from. Could cascade
into confusing failure messages or wrong behavior.

**Observed.** At end of trial:

| Source | session-1.{cloudflare,synology,tailscale} |
|---|---|
| spawn-tree.json | present (4 entries total) |
| git worktree list | absent |
| git branch | absent |
| .git/logs/refs/heads/bosun/session-1.* | absent (refs fully removed) |
| filesystem | absent |
| claims/*.json | absent |

**Reproduction is unclear.** No bosun command was knowingly run that
should have removed the sub-worktrees while leaving spawn-tree
intact. Possibilities:

- Concurrent `git worktree remove` from outside bosun (the launched
  sub-agents could have run it themselves before exiting)
- Spotlight / iCloud / fileproviderd interaction with /Volumes/
  SneakerNet (same family as the SIGBUS issues from trial #2)
- The launched sub-agents called something destructive that removed
  their worktrees but didn't update spawn-tree

**Root cause for the bug hunt:** identify which write path can
remove a worktree without updating spawn-tree, OR add a
reconciliation step (`spawntree.Sync()`) that prunes entries with
missing worktrees.

### Bug B — parent has no signal that its subs vanished

**Severity:** High. The parent's reasoning is based on
`bosun_spawn`'s success return value; once the call returns OK, the
parent treats the subs as present. There's no follow-up tool
(`bosun_check` doesn't cover this), no event stream, no MCP
notification, no way for the parent to learn its subs died.

In this trial the parent silently took on the per-package work that
the subs were supposed to do. That's the worst-case outcome — it
looks like progress but violates the brief.

**Fix shape.** Either:

1. `bosun_spawn`'s return value should expose a "verify presence"
   handle the parent can re-check (per-sub heartbeat or status URL)
2. A new MCP tool `bosun_check_tree session-1` that returns the
   current state of each child (alive / dead / no-launch-attempt)
3. The PreToolUse hook (v0.9.1 #6) could surface "sub-X is in
   spawn-tree but its worktree is gone" as an inline warning before
   the parent edits per-package files

### Bug C — recipe was followed but merge --tree was skipped

**Severity:** Low. The agent reasonably skipped step 6 once it had
done all the per-package work itself — there was nothing to merge.
But the brief said "When all three subs are DONE, run `bosun merge
--tree session-1`." Subs were never DONE (they never existed long
enough). The agent had to make a judgment call.

The recipe form could harden this with a verification step:

> 6. **Verify** all three subs are DONE before merging. If any
>    sub's state is not DONE, STOP and ask the operator before
>    proceeding solo.

That puts the agent on the hook for noticing the divergence.

### Bug D — launcher silently failed (probably)

**Severity:** High. Three sub-agents should have been spawned in
Ghostty windows. Only the parent's claude process was visible in the
process table during the trial. Either:

1. The launcher returned an error and the spawn impl absorbed it as
   "soft failure" (the comment in `internal/spawn/spawn.go:174-176`
   acknowledges this — but the parent doesn't see it because
   `bosun_spawn` returns the failures in the result, not as errors)
2. The launcher succeeded, the sub-agents launched, but they exited
   immediately for some reason (Ghostty quirk on /Volumes? rapid-fire
   AppleScript? something else)
3. macOS killed the rapid-fire Ghostty spawn (System Integrity
   Protection, or some throttle)

The investigation needs to repro on a controlled setup — small in-
repo test that calls `launcher.Launch` three times in quick succession
and watches what happens to the spawned processes.

## What the recipe got right

- **Agent obeyed step 1.** Parent wrote the shared-interface file
  AND committed it before spawning. This was the v0.9 spec's
  required base for subs.
- **Agent called bosun_spawn three times.** With the correct labels,
  correct parent, correct briefs (the sub-section bodies were
  extractable from the parent's BOSUN_BRIEF.md since the brief
  parser kept them as part of session-1's body).
- **Subs branched from parent's HEAD.** All three sub-branches
  pointed to `4397a4d`, confirming the v0.9 spec's "subs inherit
  parent's HEAD" guarantee.
- **Dotted-label worktree naming worked.**
  `homelab-status-mcp-bosun-1.cloudflare` etc. — the
  `WorktreeSuffixForLabel` substitution did what it should.

The decision-UX gap from trial #3b (issue #13) is empirically closed
by the recipe template. Bugs A-D are different — they live downstream
of the spawn call, in the launch + lifecycle machinery.

## Recommendation

**Run a bug hunt round** focused on the v0.9 spawn lifecycle:

- Lane 1: `internal/spawn/` — identify when a sub-worktree can be
  removed without updating spawn-tree; add `spawntree.Sync()` or a
  reconciliation step
- Lane 2: `internal/mcp/tool_spawn.go` — design `bosun_check_tree`
  (per Bug B) and write a spec for it
- Lane 3: `internal/launcher/` — repro the rapid-fire launch failure
  in an in-repo test; identify the actual failure mode
- Lane 4: documentation — update the recipe template with the
  verification step from Bug C; update `docs/v0.9-spawn-spec.md`
  with what the trial #3c surfaced about parent visibility into sub
  lifecycle

Until these land, **do not flip v0.9 to public via Phase 4b** of the
launch checklist. The spawn surface needs Bug B fixed at minimum —
without it, every spawn-failure outcome looks like silent solo work
to the operator.
